Status: Implemented (2026-07-06).


# Text-Mark Candidate Search Roadmap

This plan covers porting the macOS/Swift plain text-mark candidate search into
the Windows TSF C++ input method. These are the candidates produced by Swift's
`Engine.searchTextMarks(for:)` and represented as `LexiconType.text`, for
examples such as `ABC`, `AirDrop`, `Face ID`, `RAM`, and `SwiftUI`.

The goal is to make exact 26-key alphabetic input return plain text candidates
from the bundled `mark_table`, merge them into the existing candidate list with
the same priority as the Swift implementation, and commit them as literal text
without Jyutping comments, character-variant conversion, or input-memory
learning.

## Source Of Truth

Use the macOS implementation as the behavior reference:

- `Modules\CoreIME\Sources\CoreIME\TextMark.swift`
  - `Engine.searchTextMarks(for:)`
  - Queries `mark_table` by `spell = hashCode(input)`.
  - Returns `Lexicon(input: text, text: mark)`.
- `Modules\CoreIME\Sources\CoreIME\Lexicon.swift`
  - `LexiconType.text`
  - Plain text initializer sets `romanization = input`, `mark = input`, and
    `number = 0`.
  - Non-Cantonese equality and hashing use only `text`.
- `Modules\CoreIME\Sources\CoreIME\Candidate.swift`
  - `.text` candidates display `text` and have no comment.
- `Modules\CoreIME\Sources\CoreIME\Converter.swift`
  - Normal dispatch order is
    `idealMemory.prefix(3) + defined + marks + idealMemory + chained`.
  - Character-standard conversion is applied only to Cantonese candidates.
- `InputMethod\JyutpingInputController.swift`
  - Normal suggestions query text marks when English suggestions are enabled.
  - Reverse lookup paths also query text marks using the full buffer, including
    the reverse lookup selector key.

Out of scope as source material:

- `Engine.queryTextMarks(for:)`, because it is the iOS 9-key path using
  `nine_key_code`.
- `Keyboard\SharedModels\DefinedLexicon.swift` and iOS keyboard behavior.
- macOS System Text Replacements. Swift maps those to `LexiconType.text`, but
  Windows needs a separate platform-specific design for user dictionary or text
  replacement integration.

## Current Windows Baseline

Relevant files:

- `Jyutping\ImeTypes.*`
  - Already defines `Ime::LexiconType::Text`.
  - Already has `Ime::Lexicon::PlainText(input, text)`.
  - Non-Cantonese equality compares only candidate `text`, matching Swift.
- `Jyutping\Resources\ime.sqlite3`
  - Already contains `mark_table`.
  - Verified schema:
    `id`, `input`, `mark`, `spell`, `nine_key_code`.
  - Verified indexes:
    `ix_mark_spell`, `ix_mark_nine_key_code`.
  - The current database has 15,302 text-mark rows.
- `Jyutping\ImeDatabase.*`
  - Does not verify `mark_table`.
  - Does not expose a text-mark query helper.
- `Jyutping\InputEngine.*`
  - `Suggest()` currently returns Cantonese lexicon suggestions plus input
    memory.
  - `MergeMemorySuggestions()` does not accept text-mark candidates.
  - No public `SearchTextMarks()` API exists yet.
- `Jyutping\CompositionProcessorEngine.*`
  - Caches one `std::vector<Ime::Lexicon>` for the current buffer.
  - Normal input calls `_inputEngine.Suggest(inputKeys)`.
  - Reverse lookup calls `_inputEngine.ReverseLookup(method, queryKeys)`, where
    `queryKeys` excludes the leading selector key.
  - Candidate display currently converts every candidate text through
    `ConvertText()` and always uses `suggestion.romanization` as the comment.
  - Memory learning already skips non-Cantonese lexicons.

## Scope

Implement these behaviors:

- Search bundled `mark_table` by exact full-buffer 26-key input.
- Return text-mark rows as `Ime::Lexicon::PlainText(input, mark)`.
- Include text marks in normal Cantonese suggestions.
- Include text marks in reverse lookup buffers by searching the full buffer,
  including the leading selector key.
- Keep text-mark candidates distinct by candidate text.
- Display text-mark candidates without a comment.
- Commit text-mark candidates as literal candidate text.
- Keep text-mark candidates out of input memory and candidate forgetting.
- Keep the Windows project dedicated to 26-key QWERTY input.

Out of scope for this roadmap:

- iOS 9-key `nine_key_code` lookup.
- New emoji or symbol candidate search.
- Windows text replacement integration.
- A new settings UI.
- Downloading, regenerating, or migrating the bundled SQLite database.
- Changing Cantonese ranking, segmentation, punctuation behavior, or reverse
  lookup algorithms beyond inserting text-mark candidates.

## Desired Behavior

Normal input:

- Typing `abc` should show text-mark candidates such as `ABC` before normal
  Cantonese suggestions, unless ideal input-memory candidates are promoted
  ahead of them.
- Typing `airdrop`, `faceid`, `ram`, or `swiftui` should show `AirDrop`,
  `Face ID`, `RAM`, and `SwiftUI` respectively.
- If multiple rows share the same input, preserve database row order before
  distinct filtering.
- The pre-edit text remains the user's input string, because plain text marks
  have `mark = input`.

Reverse lookup buffers:

- Text-mark search uses the full buffer, not only the reverse lookup query
  suffix.
- For example, in a buffer beginning with `r`, `v`, `x`, or `q`, text marks are
  searched against strings like `ram` or `qrcode`, not just `am` or `rcode`.
- If the reverse lookup suffix is empty or invalid, text-mark candidates can
  still be shown for the full buffer when the full buffer matches `mark_table`.
- Reverse lookup Cantonese candidates still use the current selector-specific
  lookup paths.

Candidate display:

- `LexiconType::Text` candidate text is displayed exactly as stored in
  `mark_table.mark`.
- `LexiconType::Text` candidates have an empty comment in the candidate window.
- Character-standard conversion applies only to Cantonese candidates.
- Input-memory learning and forgetting remain Cantonese-only.

## Implementation Phases

### Phase 1: Database Query Support

Add a text-mark query to `ImeDatabase`.

Files:

- `Jyutping\ImeDatabase.h`
- `Jyutping\ImeDatabase.cpp`

Changes:

- Add `std::vector<std::wstring> QueryTextMarksBySpell(int64_t spell) const;`.
- Add `mark_table` to `VerifySchema()`:

```sql
SELECT mark FROM mark_table WHERE spell = ? LIMIT 0;
```

- Implement the query with:

```sql
SELECT mark FROM mark_table WHERE spell = ? ORDER BY rowid;
```

- Use existing UTF-16 SQLite helpers (`ColumnText`, `Statement`, `LogError`).
- Do not expose or use `nine_key_code`.

Notes:

- The Swift SQL is `SELECT mark FROM mark_table WHERE spell = ?;`.
- `ORDER BY rowid` makes the Windows result deterministic while preserving the
  bundled table's natural order.

### Phase 2: InputEngine Text-Mark API

Expose the exact text-mark lookup from `InputEngine`.

Files:

- `Jyutping\InputEngine.h`
- `Jyutping\InputEngine.cpp`

Changes:

- Add public overloads:

```cpp
std::vector<Lexicon> SearchTextMarks(std::wstring_view input) const;
std::vector<Lexicon> SearchTextMarks(const std::vector<VirtualInputKey>& keys) const;
```

- Implement the key overload as:
  - Return empty when the engine is not prepared or the key list is empty.
  - Convert keys with `TextFromKeys(keys)`.
  - Compute `HashCode(text)`.
  - Query `_database.QueryTextMarksBySpell(hash)`.
  - Map each row to `Lexicon::PlainText(text, mark)`.
- Keep the lookup exact. Do not segment, prefix-match, slice, or tone-convert
  text-mark input.
- Use the existing `HashCode()` implementation in `ImeTypes.cpp`, which matches
  Swift's rolling 31 hash behavior.

### Phase 3: Normal Suggestion Merge

Port the Swift dispatch priority for text marks.

Files:

- `Jyutping\InputEngine.cpp`

Preferred shape:

- Replace `MergeMemorySuggestions(memory, queried)` with a helper that also
  accepts `marks`.
- Keep the existing memory split:
  - `idealMemory`
  - `notIdealMemory`
  - `chained`
- Build final output in this order:

```text
idealMemory.prefix(3)
textMarks
idealMemory
chained
```

- Run existing `Distinct()` over the final sequence.

Behavior notes:

- This intentionally matches Swift after dropping `defined`, which is the
  macOS System Text Replacements source.
- When there is no ideal memory, text marks appear before normal queried
  Cantonese suggestions.
- When ideal memory exists, up to three ideal memory candidates remain ahead of
  text marks, matching Swift.
- Text marks do not participate in Cantonese compound concatenation because
  `Concatenate()` and `JoinLexicons()` already reject non-Cantonese lexicons.

### Phase 4: Reverse Lookup Integration

Include text marks in reverse lookup candidate lists without changing the
reverse lookup algorithms.

Files:

- `Jyutping\CompositionProcessorEngine.cpp`
- `Jyutping\CompositionProcessorEngine.h` if helper declarations are added.

Changes:

- In `GetInputSuggestions()`, compute text marks from `inputKeys`, not from
  `reverseLookupQueryKeys`.
- For normal input, this can be handled entirely by `_inputEngine.Suggest()`
  once Phase 3 is complete.
- For reverse lookup:
  - Query `std::vector<Ime::Lexicon> textMarks =
    _inputEngine.SearchTextMarks(inputKeys);`.
  - Query reverse lookup suggestions only when `reverseLookupQueryKeys` is not
    empty.
  - Merge as `textMarks + reverseLookupSuggestions`, then distinct.
  - Keep the early empty-result path only when both `textMarks` and reverse
    lookup suggestions are empty.

Important details:

- Text-mark `inputCount` should equal the full buffer length.
- Existing `_InputCount` handling for reverse lookup adds the selector key back
  for displayed candidates. With full-buffer text marks, this still resolves to
  the full buffer length.
- `CandidateAt()` currently returns `std::nullopt` for reverse lookup buffers.
  That can stay unchanged for this roadmap because committing uses the candidate
  window string and memory learning should not apply to reverse lookup or text
  candidates.

### Phase 5: Candidate Presentation

Match Swift's display behavior for `.text` candidates.

Files:

- `Jyutping\CompositionProcessorEngine.cpp`
- `Jyutping\CompositionProcessorEngine.h` if helper declarations are added.

Changes:

- Update `DisplayTextForCandidate()`:
  - If `suggestion.IsCantonese()`, return converted text as today.
  - Otherwise, return `suggestion.text` unchanged.
- Add a candidate-comment helper, for example:

```cpp
std::wstring CommentTextForCandidate(const Ime::Lexicon& suggestion) const;
```

- Return an empty comment for `LexiconType::Text`.
- Keep `suggestion.romanization` as the comment for Cantonese candidates.
- Use this helper when filling `_candidateItemCommentStorage`.

Rationale:

- Swift `Candidate` sets `.text` comments to `nil`.
- `Lexicon::PlainText()` stores `romanization = input` for internal parity with
  Swift, but that input string is not meant to be shown as a candidate comment.

### Phase 6: Optional English-Suggestions Setting

The Swift code gates text marks behind `AppSettings.isEnglishSuggestionsOn`.
Windows currently has no corresponding setting.

Recommended first implementation:

- Keep text-mark search always on.
- Do not block the core port on a new settings surface.

Follow-up option:

- Add a persisted `EnglishSuggestions` setting only after there is a broader
  settings UI or language-bar story for user-configurable candidate sources.
- If added later, gate both normal and reverse lookup text-mark calls behind
  that setting.

## Testing Plan

Database-level checks:

- Verify `ImeDatabase::VerifySchema()` succeeds with the existing bundled
  database.
- Verify exact text-mark rows are returned for known inputs:
  - `abc` -> `ABC`
  - `airdrop` -> `AirDrop`
  - `faceid` -> `Face ID`
  - `ram` -> `RAM`
  - `swiftui` -> `SwiftUI`
- Verify an unknown input returns an empty vector.
- Verify lookup is exact: prefixes of `swiftui` do not return `SwiftUI`.

Engine-level checks:

- `InputEngine::SearchTextMarks(L"abc")` returns `LexiconType::Text`.
- Returned text lexicons have:
  - `text = mark_table.mark`
  - `romanization = input`
  - `input = input`
  - `inputCount = input.size()`
  - `mark = input`
  - `number = 0`
- Normal `Suggest()` includes text marks before regular Cantonese suggestions
  when no ideal input memory is present.
- Distinct filtering removes duplicate text-mark text values while preserving
  first occurrence order.
- Text marks are not passed to `InputMemory::Handle()` or `InputMemory::Forget()`.

Composition/UI checks:

- Candidate window shows plain text-mark candidates with no comment column text.
- Candidate window still shows romanization comments for Cantonese candidates.
- Selecting a text-mark candidate commits the stored mark text exactly.
- Incremental finalization works when the selected text mark consumes the full
  buffer.
- Character-variant modes do not transform plain text-mark candidates.
- Normal Cantonese suggestions still transform according to the selected
  character variant.

Reverse lookup checks:

- A reverse lookup buffer whose full text exists in `mark_table` shows the text
  mark candidate.
- Reverse lookup Cantonese results still appear after text marks.
- Selector-only buffers still behave predictably:
  - Show matching full-buffer text marks if any exist.
  - Otherwise show no candidates until a valid reverse query exists.
- Committing a text-mark candidate from a reverse lookup buffer commits literal
  text and does not write input memory.

## Acceptance Criteria

- `mark_table` is verified at engine startup.
- Exact 26-key text-mark input produces `LexiconType::Text` candidates.
- Text-mark candidates appear in normal suggestions using Swift-compatible
  priority.
- Text-mark candidates can appear in reverse lookup buffers using the full
  buffer text.
- Text-mark candidate display has no romanization/comment.
- Text-mark candidate commit inserts the candidate text exactly.
- Character-standard conversion does not mutate text-mark candidates.
- Input memory remains Cantonese-only.
- No iOS 9-key lookup code is ported.
- Existing reverse lookup, candidate navigation, candidate selection, and
  Cantonese suggestion behavior remain intact.

## Risks And Mitigations

- Risk: Text marks outrank useful Cantonese candidates too aggressively.
  - Mitigation: Follow Swift's order first; revisit only with user-visible
    examples after parity is implemented.
- Risk: Text marks in reverse lookup buffers surprise users who expect only
  reverse lookup candidates.
  - Mitigation: This is Swift parity. Gate later behind an English-suggestions
    setting if needed.
- Risk: Non-Cantonese candidates accidentally enter input memory.
  - Mitigation: Existing memory entry points already reject non-Cantonese
    lexicons; keep tests around text-mark selection.
- Risk: Plain text containing CJK characters is converted by character-standard
  logic.
  - Mitigation: Explicitly bypass conversion for non-Cantonese lexicons in
    `DisplayTextForCandidate()`.
- Risk: Schema verification fails on older copied databases missing
  `mark_table`.
  - Mitigation: This project ships `ime.sqlite3` with the DLL; document that
    text-mark support requires the current bundled database.

## Future Work

- Add a Windows equivalent of Swift's `isEnglishSuggestionsOn` setting.

- Port emoji and symbol candidate sources if a later roadmap calls for them.
