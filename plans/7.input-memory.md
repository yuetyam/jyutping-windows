Plan: Port InputMemory from the macOS/Swift implementation.

Status: Planned.


# Input Memory Roadmap

This plan covers porting the learned candidate memory behavior from the
macOS/Swift Jyutping input method into the Windows TSF C++ input method. The
goal is to remember user-selected Cantonese candidates, promote them in future
candidate lists, and allow remembered entries to be forgotten or cleared.

The migration code in the Swift implementation is explicitly out of scope. The
Windows port should create and use the current `core_memory` schema directly,
without reading old `memory` tables, migration markers, or obsolete Apple
database files.

## Source References

Use the macOS implementation as the primary source:

- `InputMethod/Models/InputMemory.swift`
- `InputMethod/Models/MemoryLexicon.swift`
- `Modules/CoreIME/Sources/CoreIME/Converter.swift`
- `InputMethod/JyutpingInputController.swift`

Use the iOS keyboard implementation only as a comparison point:

- `Keyboard/SharedModels/InputMemory.swift`

Do not port the iOS 9-key fields or search paths. This Windows project remains
a 26-key QWERTY desktop IME.

## Scope

Implement these behaviors:

- Prepare a per-user SQLite memory database.
- Store selected Cantonese lexicons by `(word, romanization)`.
- Increment frequency and update recency when an existing entry is selected.
- Insert new entries with computed shortcut and spell keys.
- Query memory alongside normal engine suggestions.
- Mark ideal memory suggestions with `number == -1`.
- Mark non-ideal memory suggestions with `number == -2`.
- Merge memory suggestions with normal suggestions in the same spirit as Swift
  `Converter.dispatch`.
- Forget the highlighted remembered candidate.
- Clear all input memory.

Out of scope:

- InputMemory migration from older Apple schemas.
- Any `InputMemoryMigration2026` state.
- Cleanup of old Apple database files or Apple `UserDefaults` keys.
- An input-memory on/off preference. Memory suggestions and learning are always
  on for this roadmap.
- iOS/keyboard 9-key columns:
  - `nine_key_anchors`
  - `nine_key_code`
- 9-key search behavior.
- Syncing memory across devices or Windows users.
- Import/export UI.
- A full preferences window, unless a later roadmap adds one.

## Current Windows Baseline

Relevant files:

- `Jyutping/ImeTypes.*`
  - Already has `Lexicon::IsInputMemory()`,
    `Lexicon::IsIdealInputMemory()`, and
    `Lexicon::IsNotIdealInputMemory()`.
  - Uses `number < 0` to identify input memory.
- `Jyutping/InputEngine.*`
  - Owns the bundled lexicon database.
  - Owns `Segmenter` and `PinyinSegmenter`.
  - Produces normal Cantonese suggestions.
  - Already has much of the same segmentation and partial-match logic that
    Swift `InputMemory.search` uses.
- `Jyutping/ImeDatabase.*`
  - Opens the bundled `ime.sqlite3` read-only.
  - Provides SQLite helper patterns for prepared statements, UTF-16 text, and
    schema verification.
- `Jyutping/CompositionProcessorEngine.*`
  - Owns `Ime::InputEngine`.
  - Caches suggestions in `_cachedSuggestions`.
  - Converts display text through `CurrentCharacterStandard()`.
  - Builds `CCandidateListItem` values with display text, comment, and input
    count.
- `Jyutping/CandidateListUIPresenter.*` and `Jyutping/CandidateWindow.*`
  - Expose only the selected display string and input count today.
  - Do not expose the selected source `Ime::Lexicon`.
- `Jyutping/Jyutping.vcxproj` and `Jyutping/Jyutping.vcxproj.filters`
  - Must be updated when new source/header files are added.

Current gaps:

- No user memory database exists on Windows.
- Candidate ranking cannot learn from prior selections.
- Finalization loses source lexicon metadata after converting candidate display
  text, so memory cannot safely store selected candidates yet.
- There is no selected-candidate sequence equivalent to Swift
  `selectedSequence`.
- There is no forget or clear-memory command.

## Target Design

Add a dedicated input-memory component:

- `Jyutping/InputMemory.h`
- `Jyutping/InputMemory.cpp`

Recommended public API:

```cpp
namespace Ime {

class InputMemory
{
public:
    bool Prepare();
    bool IsPrepared() const;

    void Close();
    bool Handle(const Lexicon& lexicon);
    bool Forget(const Lexicon& lexicon);
    bool DeleteAll();

    std::vector<Lexicon> Suggest(
        const std::vector<VirtualInputKey>& keys,
        const Segmentation& segmentation) const;

private:
    // SQLite connection and helpers.
};

} // namespace Ime
```

Use a separate SQLite database from the bundled immutable lexicon database:

```text
%LOCALAPPDATA%\Jyutping\memory.sqlite3
```

Open it with:

```cpp
SQLITE_OPEN_READWRITE | SQLITE_OPEN_CREATE | SQLITE_OPEN_FULLMUTEX
```

Set a short busy timeout after opening so multiple TSF host processes can avoid
failing immediately under normal SQLite file locking.

Keep the component independent from TSF UI classes. `InputMemory` should depend
on engine types such as `Lexicon`, `VirtualInputKey`, and `Segmentation`, not on
`CJyutping`, candidate windows, or compartments.

## Memory Schema

Create only the current schema:

```sql
CREATE TABLE IF NOT EXISTS core_memory (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    word TEXT NOT NULL,
    romanization TEXT NOT NULL,
    frequency INTEGER NOT NULL,
    latest INTEGER NOT NULL,
    shortcut INTEGER NOT NULL,
    spell INTEGER NOT NULL,
    UNIQUE (word, romanization)
);
```

Create these indexes:

```sql
CREATE INDEX IF NOT EXISTS ix_core_memory_frequency
    ON core_memory (frequency);

CREATE INDEX IF NOT EXISTS ix_core_memory_shortcut
    ON core_memory (shortcut, frequency DESC);

CREATE INDEX IF NOT EXISTS ix_core_memory_spell
    ON core_memory (spell, frequency DESC);

CREATE INDEX IF NOT EXISTS ix_core_memory_strict
    ON core_memory (spell, shortcut, frequency DESC);
```

Important compatibility detail:

- The memory `shortcut` field should follow Swift `MemoryLexicon`:
  `HashCode(anchorText)`, where `anchorText` is the first letter of each
  romanization syllable.
- The memory `spell` field should follow Swift `MemoryLexicon`:
  `HashCode(letterText)`, where `letterText` is the romanization with only
  lowercase Latin letters.
- Do not reuse the bundled lexicon database's `anchors` encoding for memory
  rows unless the query logic is also deliberately changed. The Swift memory
  database uses hash codes for both `shortcut` and `spell`.

## Phase 1: Add Storage Foundation

1. Add `InputMemory.h` and `InputMemory.cpp`.
2. Add both files to:
   - `Jyutping/Jyutping.vcxproj`
   - `Jyutping/Jyutping.vcxproj.filters`
3. Implement path creation:
   - Locate `%LOCALAPPDATA%`.
   - Create `%LOCALAPPDATA%\Jyutping` if needed.
   - Open `memory.sqlite3` inside that folder.
4. Implement SQLite helpers:
   - RAII statement wrapper.
   - UTF-16 bind/read helpers.
   - `Prepare(sql, statement)` with logging.
   - `CurrentTimeMilliseconds()`.
5. Implement `Prepare()`:
   - Open database.
   - Create table.
   - Create indexes.
   - Return `false` if any required setup fails.
6. Do not implement migration hooks, migration settings, or old-table probes.

Acceptance criteria:

- `InputMemory::Prepare()` creates a fresh database and schema.
- Calling `Prepare()` repeatedly is harmless.
- Missing local-app-data directories are created.
- Database failures are logged but do not crash the IME process.

## Phase 2: Store, Forget, And Clear Entries

Port the candidate handling methods:

- Swift `handle(_:)`
- Swift `find(word:romanization:)`
- Swift `update(id:frequency:)`
- Swift `insert(entry:)`
- Swift `forget(_:)`
- Swift `deleteAll()`

Windows behavior:

1. `Handle(const Lexicon&)`
   - Ignore non-Cantonese lexicons.
   - Find by exact `word` and `romanization`.
   - If found, increment `frequency` and update `latest`.
   - If not found, insert a new row with `frequency = 1` and current `latest`.
2. `Forget(const Lexicon&)`
   - Ignore non-Cantonese lexicons.
   - Delete by exact `word` and `romanization`.
3. `DeleteAll()`
   - Delete all rows from `core_memory`.

Use helper logic equivalent to Swift `MemoryLexicon`:

- `word = lexicon.text`
- `romanization = lexicon.romanization`
- `shortcut = HashCode(anchorText)`
- `spell = HashCode(LatinLetterOnly(romanization))`

Add a helper for the anchor text:

```cpp
std::wstring RomanizationAnchorText(std::wstring_view romanization);
```

It should split romanization on spaces and append the first character of each
non-empty syllable.

Acceptance criteria:

- Selecting the same candidate twice stores one row with `frequency = 2`.
- `latest` changes on update.
- `Forget()` removes only the exact `(word, romanization)` pair.
- `DeleteAll()` leaves the schema intact.

## Phase 3: Query Memory Suggestions

Port the macOS 26-key search behavior from `InputMethod/Models/InputMemory.swift`.

Core query helpers:

- `ShortcutMatch(text, input, limit)`
  - Query by `shortcut = HashCode(text with y replaced by j)`.
  - Order by `frequency DESC`.
- `SpellMatch(text, input, mark, limit)`
  - Query by `spell = HashCode(text)`.
  - Order by `frequency DESC`.
- `StrictMatch(spell, shortcut, input, mark, limit)`
  - Query by both `spell` and `shortcut`.
  - Order by `frequency DESC`.
- `Perform(scheme, limit)`
  - Use `SchemeOriginText(scheme)` for `spell`.
  - Use `SchemeOriginAnchorsText(scheme)` for `shortcut`.

Internal memory row model:

```cpp
struct MemoryLexicon
{
    std::wstring word;
    std::wstring romanization;
    int64_t frequency = 0;
    int64_t latest = 0;
    std::wstring input;
    size_t inputCount = 0;
    std::wstring mark;
};
```

Sorting rules:

1. `MemoryLexicon` order:
   - Longer `inputCount` first.
   - Higher `frequency` first.
   - Newer `latest` first.
2. `RegularSorted(isOrdered)`
   - If `isOrdered` is false, create `frequencyPreferred` by sorting by
     frequency descending.
   - If `isOrdered` is true, preserve incoming order as `frequencyPreferred`.
   - Create `datePreferred` by sorting by latest descending.
   - Return distinct entries from:
     `frequencyPreferred.first(3) + datePreferred.first(5) + frequencyPreferred`.
3. `PeculiarSorted()`
   - Group by distinct `inputCount`, descending.
   - Apply `RegularSorted()` inside each group.

Return `Lexicon::Cantonese(..., number = -1)` for ideal memory suggestions and
`number = -2` for non-ideal memory suggestions.

## Phase 4: Port Input Filtering

Port the four Swift `InputMemory.suggest` cases:

1. No apostrophe, no tone input:
   - Search directly with the provided keys and segmentation.
2. Apostrophe and tone input:
   - Search syllable-letter keys.
   - Convert tone inputs with `ToneConverted`.
   - Keep candidates whose romanization is compatible with the converted input.
3. Tone input without apostrophe:
   - Search syllable-letter keys.
   - Apply the Swift tone-count matching rules.
   - Preserve the original input text through `Lexicon::ReplacedInput`.
4. Apostrophe without tone input:
   - Search syllable-letter keys.
   - Apply the Swift separator/syllable matching rules.
   - Preserve the original input text through `Lexicon::ReplacedInput`.

Do not assume the current `InputEngine::Dispatch` filters are enough for memory.
The memory query surface is separate from bundled lexicon queries and should be
verified against the Swift `InputMemory.suggest` cases.

Acceptance criteria:

- Memory suggestions work for normal full syllable input.
- Memory suggestions work for shortcut/anchor input.
- Apostrophe-separated input can match remembered multi-syllable entries.
- Tone-key input can narrow remembered candidates.
- Partial matches return `number == -2`.

## Phase 5: Merge Memory With Engine Suggestions

There are two viable integration points:

1. `InputEngine::Suggest`
   - Add `InputMemory _inputMemory`.
   - Prepare memory next to the bundled database.
   - Query memory and bundled suggestions together.
   - Return one merged `std::vector<Lexicon>`.
2. `CompositionProcessorEngine::GetInputSuggestions`
   - Keep `InputEngine` focused on bundled suggestions.
   - Add `InputMemory` to `CCompositionProcessorEngine`.
   - Use `InputEngine::Segment()` for memory queries.
   - Merge before caching `_cachedSuggestions`.

Prefer option 1 if the memory feature should be treated as part of the input
engine contract. Prefer option 2 if the TSF-specific selection lifecycle would
otherwise make `InputEngine` too stateful.

The merge should follow Swift `Converter.dispatch` behavior:

1. Split memory into:
   - `idealMemory`: `number == -1`
   - `notIdealMemory`: `number == -2`
2. Start with normal queried suggestions.
3. If ideal memory exists, remove compound queried suggestions from the initial
   queried chain.
4. Insert non-ideal memory entries ahead of the first queried candidate whose
   `inputCount` is less than or equal to the memory entry.
5. Prefix up to three ideal memory candidates.
6. Append all ideal memory candidates again before the queried chain.
7. Deduplicate by existing `Lexicon` equality.

Windows does not currently have the same `defined`, `marks`, and `symbols`
sources as the Swift converter path. Keep the merge scoped to memory plus normal
engine suggestions unless those sources are added separately.

Acceptance criteria:

- A frequently selected remembered candidate appears ahead of normal queried
  candidates for the same input.
- Ideal memory candidates are promoted more aggressively than partial memory
  candidates.
- Duplicate `(text, romanization)` pairs appear only once.
- Character-standard conversion still applies only at display time through
  `DisplayTextForCandidate`.

## Phase 6: Record Selected Candidates

Do not store the selected display string. Display text may have been converted
by `CurrentCharacterStandard()`, so storing it would corrupt the original
traditional memory entry.

Add a way to retrieve the selected source `Ime::Lexicon`:

- Add selected-index access to `CCandidateListUIPresenter` and/or
  `CCandidateWindow`.
- Add a `CCompositionProcessorEngine` helper that maps the selected candidate
  index back to `_cachedSuggestions`.
- Return `std::optional<Ime::Lexicon>` or a `bool` plus output parameter.

Record only normal Cantonese candidate selections:

- Do not record reverse lookup selections.
- Do not record plain text, emoji, symbols, or composed punctuation.
- Do not record partial character-selection actions if Windows later adds those
  shortcuts.

Port the Swift `selectedSequence` behavior:

1. Keep a selected-memory sequence during a normal input buffer.
2. Append selected Cantonese lexicons as candidates are committed.
3. If a selection consumes the whole input buffer, join the sequence with the
   final selected lexicon and call `InputMemory::Handle(joined)`.
4. If a selection leaves a tail buffer, keep the sequence and continue.
5. Clear the sequence on:
   - reverse lookup;
   - non-Cantonese selection;
   - raw text commit;
   - cancellation;
   - composition termination without a normal remembered selection.

This can be implemented in either `CJyutping` or
`CCompositionProcessorEngine`. Keeping it in `CCompositionProcessorEngine` is
likely cleaner because that class already owns raw input, cached suggestions,
and input-engine state.

Acceptance criteria:

- Selecting a full candidate records that candidate.
- Selecting multiple pieces from one buffer records the joined Cantonese phrase,
  matching Swift behavior.
- Candidate display conversion does not affect stored `word`.
- Reverse lookup selections are not remembered.

## Phase 7: Forget And Clear Memory

Forget current candidate:

1. Add a TSF key path for the Windows equivalent of macOS
   Control+Shift+Backspace/Delete.
2. When a candidate list is open and the highlighted candidate is Cantonese:
   - Resolve the highlighted source `Lexicon`.
   - Call `InputMemory::Forget(lexicon)`.
   - Refresh suggestions for the current input.
3. If the highlighted candidate was not from memory but has the same
   `(text, romanization)` as a memory row, deleting by exact pair is still safe.

Clear all memory:

1. Expose `InputMemory::DeleteAll()` through a future maintenance command or
   preferences surface.
2. Keep the API ready even if the first implementation has no UI entry point.
3. Refresh active suggestions after clearing if a composition is open.

Acceptance criteria:

- Forget removes the selected remembered entry.
- The candidate list refreshes without requiring the user to restart the IME.
- Clear memory deletes all rows but keeps the database usable.

## Phase 8: Always-On Policy

Input memory is always enabled in this roadmap. Do not add a registry value,
compartment, settings model field, language-bar toggle, or preferences UI for
turning it on or off.

Use input memory unconditionally once the memory database is prepared:

- Query memory suggestions for normal Jyutping input.
- Record normal Cantonese selections through `InputMemory::Handle()`.
- Keep `Forget()` and `DeleteAll()` available for deleting stored data.

If the memory database cannot be opened or prepared, treat memory as empty for
that process activation and continue serving normal bundled suggestions.

Acceptance criteria:

- No new input-memory setting is added to `ImeSettings` or the registry.
- Memory promotion and learning run without checking a user setting.
- Existing memory rows remain on disk until explicitly forgotten or cleared.

## Phase 9: Testing And Verification

Build:

```powershell
msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=x64
msbuild Jyutping.sln /p:Configuration=Release /p:Platform=x64
msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=ARM64
msbuild Jyutping.sln /p:Configuration=Release /p:Platform=ARM64
```

Functional checks:

- Fresh install creates `%LOCALAPPDATA%\Jyutping\memory.sqlite3`.
- Typing and selecting a Cantonese candidate inserts one memory row.
- Re-selecting the same candidate increments frequency.
- A remembered candidate appears earlier on the next matching input.
- Recent entries break frequency ties.
- Apostrophe input can match a remembered multi-syllable phrase.
- Tone input can narrow remembered candidates.
- Selecting reverse lookup candidates does not create memory rows.
- Forget removes the highlighted candidate's memory row.
- Clear all memory empties `core_memory`.
- No input-memory setting is required for suggestions or learning.

Database checks:

```sql
.schema core_memory
SELECT word, romanization, frequency, latest, shortcut, spell
FROM core_memory
ORDER BY frequency DESC, latest DESC;
```

Regression checks:

- Candidate window still shows converted display text for the current character
  variant setting.
- Candidate comments still show romanization.
- Incremental candidate finalization still preserves tail input.
- The IME remains usable if memory database opening fails.
- Multiple host processes do not corrupt the memory database.

## Risks And Notes

- TSF can load the IME DLL into multiple processes. Use SQLite file locking,
  full mutex mode, and a busy timeout. Avoid long transactions.
- Do not store converted display text. Always store original source lexicon
  text.
- `Lexicon::number` already carries both normal rank and memory markers. Keep
  `-1` and `-2` reserved for memory.
- The bundled `ImeDatabase` is read-only. Keep user memory in a separate
  read-write database.
- Memory query `shortcut` uses hash codes, while bundled lexicon `anchors` uses
  a different encoding. Mixing those encodings will make shortcut memory lookup
  fail.
- A missing memory database should be treated like an empty memory, not as an
  activation failure.
- Implement storage and query logic before UI affordances. The core behavior can
  be verified with selection learning and SQLite inspection first.
