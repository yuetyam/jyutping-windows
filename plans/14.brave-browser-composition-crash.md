Status: Completed and verified in Brave with Jyutping v0.3.1 (2026-07-16).

## Implementation Result

Implemented on 2026-07-16:

- Composition startup now uses the caller's existing write edit cookie and no
  longer requests a nested synchronous edit session.
- Startup failures return their actual `HRESULT`, do not add the first input
  key, and cannot reach an unchecked `_pComposition` dereference.
- Key-handler request and edit-session results are tracked separately and
  logged only on failure.
- Partial candidate-window creation is cleaned up and reported as failure;
  active context views are released on both candidate-window and text-extent
  paths.
- Debug x64 and Release x64, Win32, and ARM64EC builds pass with zero warnings.

Runtime verification completed on 2026-07-16:

- The affected user tested Jyutping v0.3.1 in Brave and reported that it no
  longer crashes.
- The supplied `Jyutping-v0.3.1.log` is cumulative and still contains the old
  v0.3.0 crash session at `13:22`; those earlier entries are not a regression.
- The v0.3.1 registration begins at `17:05:06`. The tested host session, PID
  `48500`, activates successfully at `17:06:22`, records none of the new
  composition, edit-session, or candidate-start failure diagnostics, and has
  no subsequent terminating detach in the supplied log.
- A crash dump is no longer required for closure. If the problem recurs, a new
  Application Error record or dump should be captured because neither log can
  identify an exception address or stack by itself.


# Brave Browser Composition Crash Roadmap

This plan covers diagnosing and fixing a reported crash when the user starts
typing with the Jyutping IME in Brave browser. Chrome and Edge reportedly do
not reproduce the crash.

The current evidence points to an IME composition-lifecycle bug exposed by a
difference in Brave's TSF text store, not to Brave's privacy or sandbox policy
as the root cause. The implementation must remain application-neutral: do not
detect Brave by executable name or add a Brave-specific compatibility mode.

## Reported Evidence

The supplied `Jyutping-v0.3.0.log` shows the relevant Brave process, PID `59732`, at
`2026-07-16 13:22:33`:

- `DllMain`, `ActivateEx`, `CompositionProcessorEngine`, `ImeDatabase`, and
  `InputMemory` all complete successfully.
- The final activation message is `_AddTextProcessorEngine: existing engine
  matches current language profile`.
- At `13:22:41`, the DLL receives `DLL_PROCESS_DETACH` with
  `processTerminating=1`.
- There is no normal `Deactivate` and no first-key, edit-session, composition,
  or candidate-window trace between activation and process termination.
- The log contains no exception code, failing address, module offset, or stack
  trace, so it confirms process termination but cannot identify the exact
  crashing instruction by itself.

Chrome and Edge not reproducing the problem does not make the current code
safe. A TSF client may legally differ in when it grants edit sessions, whether
it permits a nested synchronous request, what context view it exposes, and
whether it supplies an owner window for candidate UI.

## Current Windows Baseline

The first composing key currently follows this path:

```text
CJyutping::OnKeyDown
  -> CJyutping::_InvokeKeyHandler
  -> ITfContext::RequestEditSession(TF_ES_ASYNCDONTCARE | TF_ES_READWRITE)
  -> CKeyHandlerEditSession::DoEditSession(TfEditCookie ec)
  -> CKeyStateComposing::HandleKeyInput
  -> CJyutping::_HandleCompositionInput(ec, context, inputKey)
  -> CJyutping::_StartComposition(context)
  -> a second RequestEditSession(TF_ES_SYNC | TF_ES_READWRITE)
  -> CStartCompositionEditSession::DoEditSession
```

Relevant current behavior:

- `CKeyHandlerEditSession::DoEditSession()` already owns a valid write edit
  cookie and passes it to `_HandleCompositionInput()`.
- `_HandleCompositionInput()` discards that opportunity and calls
  `_StartComposition()`, which requests a nested synchronous write edit
  session.
- `_StartComposition()` ignores both the return value from
  `RequestEditSession()` and the edit-session result written through its final
  argument.
- `CStartCompositionEditSession::DoEditSession()` returns `S_OK` when
  `QueryInterface`, `InsertTextAtSelection`, or `StartComposition` fails, so
  the caller cannot distinguish failure from success.
- `_HandleCompositionInput()` then calls `_pComposition->GetRange(...)`
  without verifying that `_pComposition` was created.

If Brave rejects the nested synchronous request, or if composition creation
otherwise fails, `_pComposition` remains null and the next dereference can
cause an immediate access violation inside the browser process. This is the
leading hypothesis to confirm and fix.

There are two additional robustness gaps near the first-key path:

- `_InvokeKeyHandler()` uses one `HRESULT` variable for both the return value
  of `RequestEditSession()` and its edit-session result output. The output is
  overwritten by the function return, preventing useful diagnosis.
- `CCandidateWindow::_Create()` returns `TRUE` when main, shadow, or scrollbar
  window creation fails. `MakeCandidateWindow()` therefore cannot reliably
  stop candidate UI setup after a partial creation failure.

## Goals

- Never dereference `_pComposition` unless composition creation succeeded.
- Do not request a nested edit session when the caller already has a valid
  write edit cookie.
- Preserve the first input key until the composition is ready to receive it.
- Return meaningful failures through the composition and key-handler paths.
- Leave the IME in a consistent idle state after a rejected edit session,
  unsupported context, or candidate-window failure.
- Capture enough failure-only diagnostics to distinguish TSF request,
  composition, text-store, layout, and candidate-window failures.
- Verify the fix in Brave without regressing Chrome, Edge, native editors,
  Office, or the existing candidate lifecycle.

## Out Of Scope

- Browser-name detection or per-browser behavior switches.
- Relaxing Brave privacy, sandbox, or process-isolation settings.
- Replacing the TSF architecture or moving the candidate UI out of process.
- Redesigning candidate-window appearance, layout, or navigation.
- Catching access violations with structured exception handling as a substitute
  for fixing invalid state.
- Treating `processTerminating=1` as proof of a particular exception without a
  crash dump or event record.

## Phase 1: Reproduce And Capture The Failure

1. Record the exact environment:
   - Windows version and build.
   - Brave version and architecture.
   - Jyutping DLL version, architecture, and build configuration.
   - The page and field type used to reproduce, including whether it is an
     address bar, plain input, content-editable element, or password field.
   - The first key pressed and whether Cantonese mode was already enabled.

2. Reproduce in a minimal matrix before changing behavior:
   - Brave address bar.
   - Brave plain `<input>` and `<textarea>` fields.
   - Brave `contenteditable` field.
   - The equivalent Chrome and Edge fields.
   - Notepad as a native TSF baseline.

3. Capture one Windows crash artifact:
   - Prefer a user-mode dump collected with Windows Error Reporting LocalDumps
     or ProcDump.
   - Also record the Windows Application Error event, including exception code,
     faulting module, and fault offset.
   - Load the matching Jyutping PDB and confirm whether the failing frame is
     `_HandleCompositionInput`, candidate UI creation, layout handling, or a
     different callback.

4. Add temporary boundary logging if a dump is not immediately available:
   - `_InvokeKeyHandler`: key code, key category/function, request HRESULT, and
     edit-session result.
   - `_HandleCompositionInput`: entry, whether a composition already exists,
     and the result of starting a new composition.
   - The composition helper: `QueryInterface`, `InsertTextAtSelection`,
     `StartComposition`, and `SetSelection` failures.
   - `_CreateAndStartCandidate`: document manager, composition range, layout,
     UI element, active view, owner window, and window-creation failures.

5. Keep routine keystrokes out of release logs. Retain concise failure-only
   messages after the bug is fixed; remove or gate high-volume entry/exit logs.

## Phase 2: Start Composition In The Existing Edit Session

1. Replace the nested edit-session design with a helper that consumes the
   current edit cookie. Suggested shape:

   ```cpp
   HRESULT _StartComposition(TfEditCookie ec, _In_ ITfContext* pContext);
   ```

2. Move the useful body of `CStartCompositionEditSession::DoEditSession()` into
   that helper:
   - Query `ITfInsertAtSelection`.
   - Call `InsertTextAtSelection(..., TF_IAS_QUERYONLY, ...)`.
   - Query `ITfContextComposition`.
   - Call `StartComposition()` with the existing cookie.
   - Store the returned `ITfComposition` only after success.
   - Set the selection and save the composition context.

3. Return the real `HRESULT` at every failure boundary. Do not translate
   `E_NOINTERFACE`, `TS_E_READONLY`, `TF_E_NOLOCK`, or a failed
   `StartComposition()` into `S_OK`.

4. Update both existing callers to pass their current edit cookie:
   - `_HandleCompositionInput()` in `KeyHandler.cpp`.
   - `_HandleIncrementalCandidateFinalize()` in
     `CandidateListUIPresenter.cpp`.

5. Remove `CStartCompositionEditSession` if it has no remaining callers. This
   eliminates the nested `TF_ES_SYNC | TF_ES_READWRITE` request rather than
   adding a retry around it.

6. Do not use an asynchronous fallback for this helper. The remainder of
   `_HandleCompositionInput()` must not continue until composition state exists,
   and an asynchronous callback would require separately queuing and replaying
   the input key, selection, and candidate state.

## Phase 3: Make Composition Failure Atomic

1. In `_HandleCompositionInput()`, check the result from `_StartComposition()`
   before requesting selection or accessing `_pComposition`.

2. If composition start fails:
   - Do not add the input key to `CCompositionProcessorEngine`.
   - Do not create a candidate presenter or candidate window.
   - Clear any transient candidate mode created by the current operation.
   - Return the failure to `CKeyHandlerEditSession::DoEditSession()`.

3. Add an explicit `_pComposition == nullptr` guard immediately before
   `GetRange()`. This is defense in depth for future lifecycle changes, not the
   primary fix.

4. Propagate `_HandleCompositionInputWorker()` failures instead of discarding
   its return value. If reading-string insertion or candidate startup fails,
   cleanly end partial UI state and preserve a valid composition state or cancel
   it consistently.

5. Audit every `_pComposition->...` call and document its invariant:
   - Calls inside an `_IsComposing()` branch may keep that guard.
   - Calls reachable after a start operation must check the start result.
   - Candidate conversion and display-attribute helpers must return safely if
     composition was externally terminated.

6. Make `_SaveCompositionContext()` resilient to stale state. Replace the
   debug-only `assert(_pContext == nullptr)` assumption with an explicit
   lifecycle check so release builds cannot silently overwrite or leak a saved
   context.

## Phase 4: Preserve Edit-Session Results

1. Fix `_InvokeKeyHandler()` to use separate variables:

   ```cpp
   HRESULT editSessionResult = E_FAIL;
   HRESULT requestResult = pContext->RequestEditSession(
       _tfClientId,
       pEditSession,
       TF_ES_ASYNCDONTCARE | TF_ES_READWRITE,
       &editSessionResult);
   ```

2. Interpret the two results separately:
   - `requestResult` reports whether TSF accepted the request.
   - `editSessionResult` reports the edit-session callback result when it ran;
     asynchronous acceptance may initially report `TF_S_ASYNC`.

3. Return the request failure immediately. When the callback ran
   synchronously, preserve its failure rather than overwriting it.

4. Add failure-only logs containing both HRESULT values and the key
   category/function. Avoid logging raw user-composed text.

5. Apply the same two-variable convention to other touched
   `RequestEditSession()` sites. In particular, review:
   - candidate click finalization in `CandidateListUIPresenter.cpp`;
   - layout updates in `TfTextLayoutSink.cpp`;
   - asynchronous composition termination in `EndComposition.cpp`.

6. Do not mechanically add asynchronous retries everywhere. Each caller must
   decide whether its state can safely outlive the current callback.

## Phase 5: Harden Candidate UI Failure Handling

This phase addresses a secondary first-key risk. It should not replace the
composition-lifecycle fix.

1. Correct `CCandidateWindow::_Create()` so it returns `FALSE` when creation of
   the main window, shadow window, or scrollbar fails.

2. Destroy any partially created child or owner windows before returning
   failure. The object must remain safe for `DisposeCandidateWindow()` and its
   destructor.

3. In `MakeCandidateWindow()`:
   - release the `ITfContextView` returned by `GetActiveView()`;
   - preserve the actual HRESULT from `GetActiveView()` and `GetWnd()` in
     failure logs;
   - permit a null owner window only if creating the top-level tool window with
     no owner is an intentional supported path;
   - return a window-creation failure such as `HRESULT_FROM_WIN32(GetLastError())`
     instead of reporting every failure as `E_OUTOFMEMORY`.

4. If `_StartCandidateList()` fails after `BeginUIElement()`, balance
   `EndUIElement()`, dispose partial windows, unadvise the layout sink, delete
   the presenter, and restore `_candidateMode = CANDIDATE_NONE`.

5. In `CTfTextLayoutSink::_GetTextExt()`, release `ITfContextView` on both
   success and failure. A browser that frequently rejects `GetTextExt()` must
   not leak one COM reference per layout query.

6. Treat `GetTextExt()` failure as a positioning limitation, not a browser
   crash condition. Keep the candidate UI hidden or use an existing safe
   position until a valid rectangle is available.

## Phase 6: Focused Static And Build Validation

1. Run targeted searches after implementation:
   - no remaining `_StartComposition(pContext)` calls;
   - no nested synchronous composition-start request;
   - every reachable `_pComposition` dereference has a clear invariant;
   - every touched `RequestEditSession()` has distinct request/session results.

2. Run repository hygiene checks:

   ```text
   git diff --check
   ```

3. Build on Windows with the supported configurations most relevant to browser
   hosts:

   ```powershell
   msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=x64
   msbuild Jyutping.sln /p:Configuration=Release /p:Platform=x64
   msbuild Jyutping.sln /p:Configuration=Release /p:Platform=ARM64EC
   ```

4. Build `Win32` as well because 32-bit TSF hosts use that DLL even though the
   reported Brave process is expected to be 64-bit.

5. Use compiler warnings and static analysis to catch uninitialized HRESULTs,
   COM reference leaks, and null dereferences in the touched paths.

## Phase 7: Browser And Application Regression Matrix

For each application, test an empty field, continued composition, candidate
selection, cancellation, focus changes, and repeated activation/deactivation.

Brave, Chrome, and Edge:

- Address bar where the browser permits TSF composition.
- Plain `<input>` and `<textarea>`.
- `contenteditable` field.
- First key after switching to Jyutping.
- Rapid typing immediately after focus.
- Backspace, Escape, Space, Enter, and number-key candidate selection.
- Change tabs or close the tab while a composition and candidate window are
  active.
- Verify both visible custom candidate UI and TSF-owned UI if the browser or
  Windows requests integration through `ITfUIElementMgr`.

Native and desktop compatibility:

- Notepad.
- Windows search or another system text field.
- Word or another application that motivated the existing asynchronous key
  handler request.
- One application that provides no usable owner `HWND` or text extent.

For each run, verify:

- the host process does not crash or restart;
- the first input key is neither lost nor duplicated;
- pre-edit text and candidates match the engine result;
- candidate selection commits exactly once;
- failed composition start leaves the field unchanged and the IME reusable;
- no stale candidate window remains after focus or context destruction;
- failure logs contain HRESULTs but no user-composed content.

## Phase 8: Diagnose Any Remaining Brave-Only Failure

If Brave still crashes after the composition fix:

1. Collect a fresh dump with matching PDBs and compare its frame with the
   original dump.

2. Temporarily disable only candidate UI creation while keeping composition
   active. If the crash disappears, isolate these boundaries in order:
   - `ITfUIElementMgr::BeginUIElement()`;
   - `ITfContext::GetActiveView()` and `ITfContextView::GetWnd()`;
   - custom `CreateWindowEx()` calls;
   - `GetTextExt()` and layout-sink callbacks;
   - DirectWrite/Direct2D initialization and painting.

3. If the crash occurs before candidate creation, isolate:
   - key classification in `_IsKeyEaten()`;
   - the outer key-handler `RequestEditSession()`;
   - `InsertTextAtSelection()`;
   - `ITfContextComposition::StartComposition()`;
   - composition display-attribute updates.

4. Fix the failing interface contract generically. Do not branch on Brave's
   executable name; the same TSF behavior may occur in another Chromium fork or
   future Chrome/Edge release.

## Acceptance Criteria

- Starting to type with Jyutping in Brave no longer crashes or restarts Brave.
- `_StartComposition()` uses the caller's valid `TfEditCookie` and does not
  request a nested synchronous write edit session.
- Composition-start failures never reach an unchecked `_pComposition`
  dereference.
- The first key is processed once after successful composition start and is not
  added to engine state after failure.
- Chrome and Edge retain their current working behavior.
- Notepad and Word composition, candidate selection, cancellation, and focus
  changes remain functional.
- Candidate-window creation failures are reported accurately and leave no
  partial UI or stale candidate mode.
- Failure logs distinguish request HRESULTs from edit-session HRESULTs without
  recording user input text.
- x64, Win32, and ARM64EC builds succeed on Windows.
- A Brave regression run completes without a new Application Error event or
  crash dump attributable to `Jyutping.dll`.

## Implementation Order

Implement in this order so each stage narrows the remaining risk:

1. Capture the original crash artifact or add boundary diagnostics.
2. Refactor composition start to reuse the current edit cookie.
3. Make composition failure atomic and guard lifecycle invariants.
4. Preserve and log edit-session results correctly.
5. Harden candidate-window and text-layout failure cleanup.
6. Build all supported browser-host architectures on Windows.
7. Run the Brave/Chrome/Edge and native-application regression matrix.
8. Remove high-volume diagnostic logging after the failure is confirmed fixed.
