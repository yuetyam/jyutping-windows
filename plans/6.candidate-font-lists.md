# Candidate Font Lists Roadmap

This plan covers adding font-family fallback lists for candidate number labels
and candidate comment text, following the existing `CANDIDATE_FONT_NAMES`
pattern. It does not change candidate ranking, candidate paging, row layout,
color theme behavior, or TSF candidate data.

## Scope

Implement separate font lists for these candidate-window text roles:

- Candidate text: keep using `CANDIDATE_FONT_NAMES`.
- Number labels: replace the single `NUMBER_LABEL_FONT_NAME` dependency with a
  list-based fallback.
- Comment text: add a dedicated list-based fallback instead of reusing the
  number-label text format.

Out of scope for this roadmap:

- Adding a settings UI for font customization.
- Reading font lists from registry or config files.
- Changing candidate/comment font sizes.
- Changing candidate-window colors, spacing, row height, selection behavior, or
  scroll behavior.
- Porting macOS font preference UI or platform-specific font discovery.

## Current Windows Baseline

Relevant files:

- `Jyutping/Define.h` defines:
  - `CANDIDATE_FONT_NAMES`
  - `CANDIDATE_FONT_SIZE`
  - `NUMBER_LABEL_FONT_NAME`
  - `NUMBER_LABEL_FONT_SIZE`
- `Jyutping/Globals.cpp` builds `Global::candidateFontNames` and
  `Global::pDWriteFontFallback` from `CANDIDATE_FONT_NAMES`.
- `Jyutping/Globals.h` exposes the candidate font fallback globals.
- `Jyutping/CandidateWindow.h` owns:
  - `_pDWriteTextFormat` for candidate strings.
  - `_pDWriteNumberFormat` for number labels.
- `Jyutping/CandidateWindow.cpp`:
  - applies `Global::pDWriteFontFallback` only to candidate text;
  - creates number labels with `NUMBER_LABEL_FONT_NAME`;
  - measures and renders comments with `_pDWriteNumberFormat`, so comments
    currently inherit the number-label font family and size.

Current gaps:

- Number labels have only one configured family and no explicit fallback list.
- Comments do not have a dedicated font list or text format.
- The global DirectWrite fallback resource is role-specific in behavior but
  named generically as `pDWriteFontFallback`.
- Measurement and rendering paths are coupled because comments reuse the number
  label format.

## Target Design

Use one font-list constant and one DirectWrite fallback object per text role:

- Candidate text:
  - `CANDIDATE_FONT_NAMES`
  - `Global::candidateFontNames`
  - `Global::pDWriteCandidateFontFallback`
- Number labels:
  - `NUMBER_LABEL_FONT_NAMES`
  - `Global::numberLabelFontNames`
  - `Global::pDWriteNumberLabelFontFallback`
- Comments:
  - `COMMENT_FONT_NAMES`
  - `Global::commentFontNames`
  - `Global::pDWriteCommentFontFallback`

Use a dedicated DirectWrite text format in `CCandidateWindow` for each role:

- `_pDWriteTextFormat` for candidate strings.
- `_pDWriteNumberFormat` for number labels.
- `_pDWriteCommentFormat` for comments.

Keep the existing font sizes unless a later visual decision changes them:

- Candidate strings use `CANDIDATE_FONT_SIZE`.
- Number labels use `NUMBER_LABEL_FONT_SIZE`.
- Add a new `COMMENT_FONT_SIZE` for comment texts, same value as the `NUMBER_LABEL_FONT_SIZE` at this moment.

## Proposed Default Lists

Start conservatively so the visible rendering stays close to the current UI.

Candidate text remains unchanged:

```cpp
#define CANDIDATE_FONT_NAMES L"Shanggu Sans", L"IBM Plex Sans TC", L"Microsoft JhengHei UI", L"Microsoft YaHei UI", L"MiSans L3"
```

Number labels should prefer fonts with stable digit metrics:

```cpp
#define NUMBER_LABEL_FONT_NAMES L"JetBrains Mono", L"Consolas", L"Segoe UI", L"Microsoft JhengHei UI", L"Microsoft YaHei UI"
```

Comments should prefer fonts that cover Latin Jyutping, tone numbers, and CJK
when reverse lookup or future annotations need it:

```cpp
#define COMMENT_FONT_NAMES L"Inter", L"Segoe UI", L"Shanggu Sans", L"Microsoft JhengHei UI", L"Microsoft YaHei UI"
```

Keep `NUMBER_LABEL_FONT_NAME` only as a temporary compatibility alias if it
reduces churn during implementation. The final code should avoid using it in
new DirectWrite creation paths.

## Phase 1: Constants

1. Update `Jyutping/Define.h`.

2. Add:
   - `NUMBER_LABEL_FONT_NAMES`
   - `COMMENT_FONT_NAMES`
   - `COMMENT_FONT_SIZE`

3. Keep `NUMBER_LABEL_FONT_SIZE` unchanged.

4. Decide whether to remove `NUMBER_LABEL_FONT_NAME` immediately or keep it as
   a compatibility alias for one change. If kept, make sure no candidate-window
   code still depends on it after the migration.

## Phase 2: Global Font Arrays

1. Update `Jyutping/Globals.cpp`.

2. Keep the existing candidate font array but rename the fallback object for
   clarity:
   - from `pDWriteFontFallback`
   - to `pDWriteCandidateFontFallback`

3. Add arrays and counts:
   - `numberLabelFontNames`
   - `numberLabelFontNamesCount`
   - `commentFontNames`
   - `commentFontNamesCount`

4. Update `Jyutping/Globals.h` with matching extern declarations.

5. If the fallback rename is too noisy for one patch, keep
   `pDWriteFontFallback` for candidate text and add only the new role-specific
   fallbacks. Prefer the rename if only `CandidateWindow.cpp` uses it.

## Phase 3: DirectWrite Fallback Creation

1. Refactor `Global::InitDirectWrite()` in `Jyutping/Globals.cpp`.

2. Add a small helper with local linkage, for example:

```cpp
static HRESULT CreateFontFallback(
    _In_reads_(fontNamesCount) const LPCWSTR* fontNames,
    _In_ UINT32 fontNamesCount,
    _COM_Outptr_ IDWriteFontFallback** fontFallback
);
```

3. The helper should:
   - create an `IDWriteFontFallbackBuilder`;
   - map the full Unicode range `{ 0x0000, 0x10FFFF }`;
   - use the provided font list and count;
   - return the result of `CreateFontFallback`.

4. Use the helper to initialize:
   - `pDWriteCandidateFontFallback`
   - `pDWriteNumberLabelFontFallback`
   - `pDWriteCommentFontFallback`

5. Preserve current behavior where `InitDirectWrite()` returns `TRUE` once the
   DirectWrite factory is available, even if custom fallback creation fails.
   Candidate-window format creation should tolerate missing fallback objects.

6. Update `Global::UninitDirectWrite()` to release all fallback objects before
   releasing `pDWriteFactory`.

## Phase 4: Candidate Window Formats

1. Update `Jyutping/CandidateWindow.h`.

2. Add:

```cpp
ComPtr<IDWriteTextFormat1> _pDWriteCommentFormat;
```

3. Consider changing `_pDWriteNumberFormat` from `IDWriteTextFormat` to
   `IDWriteTextFormat1` so it can receive a custom font fallback. If the
   project needs to avoid that change, create it as `IDWriteTextFormat`, call
   `.As(&numberFormat1)`, and set fallback on the converted interface.

4. Update the `WM_CREATE` DirectWrite setup in `Jyutping/CandidateWindow.cpp`.

5. Create candidate text with the first candidate font-list entry as the base
   family, then apply `Global::pDWriteCandidateFontFallback`.

6. Create number labels with the first number-label font-list entry as the base
   family, then apply `Global::pDWriteNumberLabelFontFallback`.

7. Create comments with the first comment font-list entry as the base family,
   then apply `Global::pDWriteCommentFontFallback`.

8. Set paragraph alignment consistently on all three formats:
   - candidate text: center, as today;
   - number labels: center, as today;
   - comments: center initially, matching current rendering.

9. Keep DPI scaling behavior unchanged.

## Phase 5: Measurement And Rendering

1. Update `CCandidateWindow::_ResizeWindow()`.

2. Continue measuring candidate strings with `_pDWriteTextFormat`.

3. Continue measuring number labels with `_pDWriteNumberFormat`.

4. Change comment measurement to use `_pDWriteCommentFormat`.

5. Update `CCandidateWindow::_DrawList()`.

6. Continue drawing number labels with `_pDWriteNumberFormat`.

7. Continue drawing candidate strings with `_pDWriteTextFormat`.

8. Draw comments with `_pDWriteCommentFormat`.

9. Keep the row-width formula unchanged:

```text
LeftPadding | NumberLabel | NumberSpacing | CandidateWord | CommentSpacing | Comment | RightPadding
```

10. Only add `CANDIDATE_COMMENT_SPACING` when a comment exists. The current
    draw path advances by comment spacing before checking comment length; fix
    that while changing comment formatting if the visual output shows extra
    trailing spacing on rows without comments.

## Phase 6: Naming And Compatibility Sweep

1. Search for:
   - `NUMBER_LABEL_FONT_NAME`
   - `NUMBER_LABEL_FONT_NAMES`
   - `COMMENT_FONT_NAMES`
   - `pDWriteFontFallback`
   - `_pDWriteNumberFormat`

2. Ensure all measurement and drawing paths use the intended role-specific
   format.

3. Ensure no comment code still depends on number-label fonts except for the
   shared font size.

4. Keep naming consistent with existing globals:
   - lower camel case for global array names;
   - leading underscore for new private members;
   - descriptive names over abbreviations.

## Phase 7: Verification

Build verification:

```powershell
msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=ARM64
msbuild Jyutping.sln /p:Configuration=Release /p:Platform=ARM64
msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=x64
msbuild Jyutping.sln /p:Configuration=Release /p:Platform=x64
```

Manual rendering verification:

- Normal Jyutping candidates render candidate words with the existing candidate
  font fallback list.
- Number labels render with the number-label list, preferably using stable
  digit widths.
- Reverse lookup comments render with the comment list.
- Candidate rows with no comments do not allocate visible extra comment width.
- Candidate rows with long comments resize the candidate window correctly.
- Highlighted rows keep readable number labels, candidate text, and comments.
- Light and dark theme colors remain unchanged.
- DPI changes recreate all three text formats and preserve scaled sizes.
- Missing optional fonts do not prevent the candidate window from rendering.

Regression checks:

- Candidate selection by number key still maps to the same candidate.
- Mouse selection still uses row hit testing correctly.
- Candidate paging and scroll bar sizing are unchanged except for width changes
  caused by more accurate comment measurement.
- UI-less candidate mode is unaffected.

## Implementation Notes

- DirectWrite fallback objects are process-wide resources and should stay in
  `Global`, matching the existing candidate fallback design.
- `CreateTextFormat()` still needs one base family name. Use the first entry in
  each role's font list so the base family and fallback list stay aligned.
- If `IDWriteTextFormat1::SetFontFallback()` is unavailable for a converted
  format, leave that specific fallback unset rather than failing candidate
  window creation.
- Keep the implementation local to `Define.h`, `Globals.*`, and
  `CandidateWindow.*` unless a future settings feature makes the font lists
  configurable.
