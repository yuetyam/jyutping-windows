# Candidate Window Translucency Note

The Git commit of this fix: `b566e1b047eb5d95f0bd8d9236b02a74994ad2ab`

This commit fixes a readability problem in the IME candidate window when Windows
uses light mode but the client application uses a dark surface. The previous
candidate window used acrylic with a fixed gray tint and a fully transparent
Direct2D clear. In that mixed-theme case, the dark client content could show
through too strongly, making the light-mode text difficult to read.

The committed behavior keeps the Direct2D clear transparent so the candidate
window still has background translucency. Readability now comes from applying a
theme-colored acrylic tint through `SetWindowCompositionAttribute`, with the
tint alpha set by `candidateWindowAcrylicAlpha`.

The important tested tradeoff is that adding a nonzero Direct2D background alpha
made the candidate window look sharp but effectively removed visible
translucency. If this area regresses, first tune the acrylic tint alpha and
theme color before reintroducing an opaque or semi-opaque Direct2D backing fill.

Potential future risk:

- `SetWindowCompositionAttribute` and `ACCENT_ENABLE_ACRYLICBLURBEHIND` are not
  documented as a stable public acrylic API, so behavior may vary across Windows
  builds.
- The contrast result still depends on the interaction between Windows app
  theme, foreground app content, DWM acrylic composition, and the selected
  candidate row color.
- Future changes to candidate text color, selection color, scrollbar painting,
  or `Global::GetCandidateWindowBackgroundColor()` should be checked in mixed
  light-Windows/dark-app and dark-Windows/light-app scenarios.
