# CharacterStandard Converter Porting Roadmap

This plan covers porting the Swift/macOS character-standard converter functions into the Windows C++ project.
It intentionally does not cover calling these converters from candidate generation, candidate UI, settings, compartments, registry persistence, language-bar UI, or runtime mode switching. Those integrations are future tasks.

## Scope

Port only the reusable conversion surface and its direct support code:

- `CharacterStandard` enum and raw-value helpers.
- Character-standard table-name mapping.
- Direct text conversion equivalent to `Converter.convert(_:to:)`.
- Character-by-character conversion equivalent to `variantMap` and `match`.
- PRC General phrase-aware conversion equivalent to `prcGeneralConvert`.
- Simplified phrase-aware conversion equivalent to `mutilatedCovert`.
- Database schema checks and lookup APIs needed by the converter.
- Focused verification for converter behavior.

Out of scope for this roadmap:

- Applying conversion to `Ime::Lexicon`, candidate text, comments, or reverse lookup results.
- Changing `CCompositionProcessorEngine`, `CCandidateListUIPresenter`, TSF candidate wrappers, or UI rendering.
- Adding options UI or persistent settings for character standards.
- Porting `.custom`, `.etymology`, or `.opencc` conversion behavior beyond Swift-compatible no-op handling.
- Replacing or regenerating the SQLite database unless the shipped database lacks required variant tables.

## Source Of Truth

Use the Swift CoreIME implementation as the behavioral source of truth:

- `Modules\CoreIME\Sources\CoreIME\CharacterStandard.swift`
- `Modules\CoreIME\Sources\CoreIME\Converter.swift`
- `Modules\CoreIME\Sources\CoreIME\Converter+PRCGeneral.swift`
- `Modules\CoreIME\Sources\CoreIME\Converter+Mutilated.swift`
- `Modules\CoreIME\Sources\CoreIME\Candidate.swift` only as reference for future integration boundaries.

Swift character standards:

- `preset = 1`
- `custom = 2`
- `inherited = 3`
- `etymology = 4`
- `opencc = 5`
- `hongkong = 6`
- `taiwan = 7`
- `prcGeneral = 8`
- `ancientBooksPublishing = 9`
- `mutilated = 51`

Swift variant table mapping:

- `inherited` -> `variant_old`
- `hongkong` -> `variant_hk`
- `taiwan` -> `variant_tw`
- `prcGeneral` -> `variant_prc`
- `ancientBooksPublishing` -> `variant_abp`
- `mutilated` -> `variant_sim`
- all other standards behave as no-op in `Converter.convert`, even though Swift's private default table name is `variant_hk`.

## Current Windows Baseline

Relevant Windows files:

- `Jyutping\ImeDatabase.*` owns the SQLite connection and query helpers.
- `Jyutping\InputEngine.*` owns IME lookup, but should not be required to use the converter in this roadmap.
- `Jyutping\ImeTypes.*` owns shared text helpers and may host small Unicode helper functions if they are broadly useful.
- `Jyutping\Jyutping.vcxproj` and `Jyutping\Jyutping.vcxproj.filters` must be updated if new source files are added.
- `Jyutping\Resources\ime.sqlite3` already contains tables like `variant_hk` with columns `source INTEGER PRIMARY KEY` and `target INTEGER NOT NULL`.

Preferred new files:

- `Jyutping\CharacterStandard.h`
- `Jyutping\CharacterStandard.cpp`

The converter should live outside TSF and candidate UI code so future tasks can apply it from the right boundary without moving logic again.

## Design Goals

- Preserve Swift raw values so future settings migration can store the same integer values.
- Keep no-op standards no-op: `preset`, `custom`, `etymology`, and `opencc` return input unchanged.
- Keep the first port as pure conversion functionality with no behavior changes in the active IME path.
- Use the existing `ImeDatabase` connection model and prepared SQLite statements.
- Bind Unicode scalar code points as integers, matching Swift's `source` and `target` table usage.
- Preserve non-ideographic characters during phrase-aware PRC and simplified conversion.
- Preserve Swift's length-specific PRC and simplified behavior: length-1 and length-2 fallback attempts table lookup for every Swift `Character`; only the length-3-or-more greedy fallback gates single-character lookup on `Character.isIdeographic`.
- Prefer explicit, testable C++ helpers over putting large conversion logic in anonymous call sites.
- Avoid introducing OpenCC or other external dependencies for this port.

## Phase 1: Schema And Data Readiness

1. Inspect `Jyutping\Resources\ime.sqlite3` for all required variant tables:
   - `variant_old`
   - `variant_hk`
   - `variant_tw`
   - `variant_prc`
   - `variant_abp`
   - `variant_sim`

2. Expand `ImeDatabase::VerifySchema()` to validate converter tables and columns.
   - Minimal no-row statement shape: `SELECT target FROM <table> WHERE source = ? LIMIT 0;`
   - Validate every required table, because the converter should fail preparation clearly if resources are stale.

3. Add an `ImeDatabase` API for single-character variant lookup.
   - Suggested shape: `std::optional<uint32_t> QueryVariantTarget(CharacterStandard standard, uint32_t source) const;`
   - Do not make multi-character conversion call an API that prepares once per character.
   - Prefer a call-scoped prepared-statement lookup object or helper that prepares `SELECT target FROM <table> WHERE source = ? LIMIT 1;` once, then resets and reuses it for the whole string conversion.

4. Decide ownership of reusable prepared statements.
   - Simple option: a local RAII statement inside conversion functions, equivalent to Swift.
   - More optimized option: a small converter class with one prepared statement per conversion call.
   - Do not add long-lived mutable statement caches unless profiling shows a need.

## Phase 2: Public Converter Surface

1. Add `enum class CharacterStandard : int` in `CharacterStandard.h` with Swift-compatible raw values.

2. Add helper functions:
   - `CharacterStandardFromRawValue(int value)` returning `Preset` for unknown values.
   - `IsMutilated(CharacterStandard standard)`.
   - `IsTraditional(CharacterStandard standard)`.
   - `IsNoOpCharacterStandard(CharacterStandard standard)`.
   - `VariantTableName(CharacterStandard standard)` as an internal implementation detail, not a UI-facing API.

3. Add a converter API that does not affect current runtime call sites.
   - Suggested shape: `std::wstring ConvertText(const ImeDatabase& database, std::wstring_view text, CharacterStandard standard);`
   - Keep the function in an `Ime` namespace or a small `CharacterStandardConverter` class, following the surrounding engine style.

4. Keep return behavior simple and Swift-compatible.
   - Empty input returns empty input.
   - No-op standards return input unchanged.
   - Missing database rows leave the original character unchanged.
   - Invalid or unsupported scalar handling leaves the original code unit sequence unchanged.

## Phase 3: Direct Variant Conversion

1. Port Swift `variantMap(text:tableName:)` for these standards:
   - `inherited`
   - `hongkong`
   - `taiwan`
   - `ancientBooksPublishing`

2. Implement character matching equivalent to Swift `match(character:statement:)`.
   - Convert a single Unicode scalar to its integer code point.
   - Bind it to `source`.
   - If a row exists, convert `target` back to a character.
   - If lookup or character construction fails, keep the original character.

3. Handle Unicode carefully on Windows.
   - `std::wstring` uses UTF-16 on Windows, so characters outside the BMP may be surrogate pairs.
   - Add a small scalar iterator/helper if needed; do not assume every logical character is one `wchar_t`.
   - The variant tables are Han-character mappings and mostly BMP, but the converter should not corrupt non-BMP input.

4. Keep punctuation, Latin text, digits, emoji, and unlisted Han characters unchanged.

## Phase 4: PRC General Conversion

1. Port `Converter+PRCGeneral.swift` phrase table into C++.
   - Store as `std::wstring_view` pairs in `CharacterStandard.cpp`, or as a local static lookup table.
   - Keep source and target strings identical to Swift.
   - Ensure source files remain UTF-8 and wide string literals preserve non-BMP characters through UTF-16 surrogate pairs.
   - Preserve the Swift typo-free behavior, not necessarily the Swift function names.

2. Port Swift behavior by input length:
   - length 0: return input.
   - length 1: convert one character through `variant_prc`.
   - length 2: phrase-table exact match first, otherwise per-character conversion.
   - length 3 or more: phrase-table exact match first, otherwise greedy transform.

3. Port greedy transform exactly.
   - Move left to right.
   - At each position, try the longest phrase first.
   - If a phrase matches, append its replacement and advance by that phrase length.
   - If no phrase matches and the current character is ideographic, convert it through `variant_prc`.
   - If no phrase matches and the current character is not ideographic, append it unchanged.

4. Add or reuse an `IsIdeographic` helper equivalent to Swift `Character.isIdeographic` for this converter's fallback decision.
   - Port the exact ranges from Swift `CommonExtensions/CharacterExtensions.swift`:
     - `0x4E00...0x9FFF`
     - `0x3400...0x4DBF`
     - `0x20000...0x2A6DF`
     - `0x2A700...0x2B73F`
     - `0x2B740...0x2B81F`
     - `0x2B820...0x2CEAF`
     - `0x2CEB0...0x2EBEF`
     - `0x30000...0x3134F`
     - `0x31350...0x323AF`
     - `0x2EBF0...0x2EE5F`
     - `0x323B0...0x33479`
     - `0x3007`
   - Keep the helper conservative outside those Swift ranges: unknown non-Han scalars should pass through unchanged.

## Phase 5: Simplified Conversion

1. Port `Converter+Mutilated.swift` phrase table into C++.
   - Keep the public enum name `Mutilated` only if matching existing project terminology is preferred; otherwise expose user-facing comments as Simplified where appropriate.
   - Preserve raw value `51`.

2. Port Swift behavior by input length:
   - length 0: return input.
   - length 1: convert one character through `variant_sim`.
   - length 2: phrase-table exact match first, otherwise per-character conversion for both characters.
   - length 3 or more: phrase-table exact match first, otherwise greedy transform.

3. Port greedy longest-match transform using the simplified phrase table.

4. Use `variant_sim` for single-character fallback.

5. Preserve non-ideographic text unchanged in fallback paths.
   - This applies to length-3-or-more greedy fallback only; length-1 and length-2 behavior should match Swift's `char2char` fallback.

## Phase 6: Build Integration Only

1. Add new files to:
   - `Jyutping\Jyutping.vcxproj`
   - `Jyutping\Jyutping.vcxproj.filters`

2. Include the converter only where needed for compilation.
   - Avoid modifying `InputEngine::Suggest`, reverse lookup methods, candidate presentation, or TSF composition flow in this roadmap.

3. Confirm the project still builds for at least the active local target.
   - Preferred smoke build: `msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=ARM64`
   - If practical, also build `Debug|x64` because x64 remains a supported target.

## Phase 7: Verification

1. Add focused converter tests if the repository gains or already has a lightweight test harness.
   - If no harness exists, add a small internal debug-only verification route only if it does not affect shipping behavior.
   - Do not build UI or settings tests for this roadmap.

2. Minimum direct variant examples:
   - For each of `variant_old`, `variant_hk`, `variant_tw`, `variant_abp`, pick one known row from the SQLite table and verify source maps to target.
   - Verify an unmapped character returns unchanged.
   - Verify Latin text and punctuation return unchanged.

3. Minimum PRC General examples from the Swift phrase table:
   - `反覆` -> `反復`
   - `沈沒成本` -> `沉没成本`
   - `項鍊` -> `項鏈`
   - A mixed string with punctuation keeps punctuation unchanged.

4. Minimum simplified examples from the Swift phrase table:
   - `反覆` -> `反复`
   - `乾乾淨淨` -> `干干净净`
   - `甚麼` -> `什么`
   - `項鍊` -> `项链`
   - `二噁英` -> `二𫫇英` to prove non-BMP replacement literals survive the C++ port.

5. No-op examples:
   - `preset`, `custom`, `etymology`, and `opencc` return the exact original string.

6. Unicode safety examples:
   - Non-BMP characters pass through unchanged unless a table explicitly maps them.
   - Surrogate pairs in input and phrase-table replacements are not split or corrupted.

## Suggested Implementation Order

1. Add `CharacterStandard` enum and no-op helpers.
2. Validate variant tables in `ImeDatabase::VerifySchema()`.
3. Add prepared-statement character lookup support.
4. Implement direct variant conversion for inherited, Hong Kong, Taiwan, and ancient-books standards.
5. Port PRC General phrase table and greedy transform.
6. Port simplified phrase table and greedy transform.
7. Add project/filter entries.
8. Build and run focused converter verification.
9. Leave runtime use of the converter to a follow-up integration roadmap/task.
