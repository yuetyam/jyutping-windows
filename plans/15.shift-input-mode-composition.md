# Shift Input Mode Composition Finalization

This note records the implementation and debugging of the Input Mode behavior
when the user presses `Shift` while a Cantonese composition is active.

The final behavior is:

1. Commit the raw input letters to the client.
2. Close the candidate UI and end the Cantonese composition.
3. Change `InputMethodMode` to `ABC`.
4. Persist the new mode and update the language-bar Input Mode icon to `A`.

The inverse toggle from ABC back to Cantonese continues to use the existing
compartment path and restores the Cantonese language-bar icon.

## Implementation Result

Implemented and runtime-verified on 2026-07-16:

- `Shift` during an active composition now requests a dedicated asynchronous
  read/write edit session.
- The edit session calls the existing raw-finalization path before changing
  the keyboard open/close compartment.
- Raw finalization writes `GetRawInputText()` to the client, clears candidate
  and input state, and terminates the TSF composition.
- Input Mode toggling remains centralized in
  `CCompositionProcessorEngine::ToggleInputMethodMode()`.
- The existing keyboard-open compartment callback continues to synchronize
  conversion mode, persistence, and the language-bar item.
- The language-bar button remains registered with the system-defined
  `GUID_LBI_INPUTMODE`, which is required for it to appear in the Windows 11
  compact input indicator.
- Icon resource IDs are consumed directly from the current resource macros
  instead of being copied into exported integer globals in `Globals.obj`.
- A full `Debug|ARM64` rebuild completed with zero warnings and zero errors.
- The user verified that raw text commit, composition termination, mode
  switching, and both language-bar icons work correctly in the deployed DLL.

## Original Failure

Before this change, the Shift preserved-key handler only toggled
`GUID_COMPARTMENT_KEYBOARD_OPENCLOSE`.

When the user had typed Cantonese letters and then pressed `Shift`:

- the keyboard-open compartment changed from Cantonese to ABC;
- the setting was persisted;
- the language-bar state notification ran;
- but the raw letters remained in an active Cantonese composition;
- and the candidate/composition state was not terminated.

The preserved-key callback does not receive a `TfEditCookie`, so it cannot
directly edit client text or end a TSF composition. Those operations must run
inside an `ITfEditSession` with read/write access to the supplied context.

## Final Runtime Flow

The active-composition Shift path is now:

```text
CJyutping::OnPreservedKey(context, input-mode-guid)
  -> validate the Shift-only preserved key
  -> request CInputMethodModeEditSession
       with TF_ES_ASYNCDONTCARE | TF_ES_READWRITE
  -> CInputMethodModeEditSession::DoEditSession(editCookie)
       -> CJyutping::_HandleCompositionFinalizeRaw(editCookie, context)
            -> CCompositionProcessorEngine::GetRawInputText()
            -> replace the composing text with the raw letters
            -> clear the candidate list and input-key buffer
            -> terminate the TSF composition
       -> CCompositionProcessorEngine::ToggleInputMethodMode(...)
            -> toggle GUID_COMPARTMENT_KEYBOARD_OPENCLOSE
            -> update the in-memory setting
            -> persist InputMethodMode
            -> notify conversion-mode and language-bar compartment sinks
```

The ordering is intentional. The mode changes only after raw finalization
succeeds. A failure to commit or terminate the composition must not silently
leave an active Cantonese composition while the live mode reports ABC.

When no composition is active, the existing preserved-key path remains
direct: it validates Shift and toggles Input Mode without requesting an edit
session.

## Edit Session Design

`CInputMethodModeEditSession` is a small edit-session implementation local to
`KeyEventSink.cpp`.

It reuses the existing `CEditSessionBase` lifetime handling:

- the context is retained for the asynchronous request;
- the text service is retained while the session is pending;
- COM release behavior remains consistent with other Jyutping edit sessions.

The request uses `TF_ES_ASYNCDONTCARE | TF_ES_READWRITE`, matching the existing
key-handler convention. The code deliberately does not require a synchronous
edit session because some TSF clients, including Microsoft Word, may reject or
defer synchronous requests made from a key callback.

The preserved key is treated as eaten when the request succeeds and the edit
session either completed successfully or returned `TF_S_ASYNC`. Request or
session failures are logged with separate HRESULT values.

## Raw Finalization Reuse

No new text insertion algorithm was introduced. The Shift path reuses
`CJyutping::_HandleCompositionFinalizeRaw()` from `KeyHandler.cpp`.

That helper already provides the required semantics:

- read the canonical raw input from `CCompositionProcessorEngine`;
- write the raw string into the client text store;
- use the composition-aware insertion path when composing;
- clear candidates and input keys through `_HandleComplete()`;
- call `_TerminateComposition()` with the active edit cookie.

Keeping this behavior in one helper prevents Shift, Enter, Space, and other
raw-finalization paths from developing different text-store behavior.

## Input Mode Toggle Reuse

The compartment mutation was extracted to
`CCompositionProcessorEngine::ToggleInputMethodMode()`.

Both paths now use the same operation:

- ordinary Shift with no active composition;
- Shift after raw composition finalization.

The helper:

1. Reads `GUID_COMPARTMENT_KEYBOARD_OPENCLOSE`.
2. Writes the inverse value.
3. Maps the new compartment value to `InputMethodMode`.
4. Updates `_settings.inputMethodMode`.
5. Saves the value through `SettingsStore`.

The existing compartment event sinks remain responsible for mirroring the
keyboard-open state into TSF conversion mode and notifying the language-bar
button.

## Language-Bar Icon Investigation

After composition finalization was fixed, runtime testing found a second issue:
the Input Mode button changed from a black Cantonese glyph to a white Cantonese
glyph when switching to ABC. It did not show `A`.

The source and resource script initially appeared correct:

```text
IDI_INPUT_MODE_CANTONESE      = 21 -> Resources/zh.ico
IDI_INPUT_MODE_CANTONESE_ALT  = 22 -> Resources/zh-alt.ico
IDI_INPUT_MODE_ABC            = 23 -> Resources/abc.ico
IDI_INPUT_MODE_ABC_ALT        = 24 -> Resources/abc-alt.ico
```

The ICO files were decoded and confirmed to contain distinct glyph masks:

- `zh*.ico`: Cantonese glyph;
- `abc*.ico`: `A` glyph;
- normal files: black foreground;
- `*-alt.ico`: white foreground.

The compiled DLL resource table also contained the correct distinct resources
at IDs 21 through 24. This ruled out corrupt ICO files and incorrect `.rc`
bindings.

## Unsuccessful Icon Hypotheses

Several plausible fixes were tested and rejected through runtime evidence.

### Notification Flags

The language-bar compartment callback originally sent:

```cpp
OnUpdate(TF_LBI_STATUS | TF_LBI_ICON)
```

Changing this to `OnUpdate(TF_LBI_ICON)` did not change runtime behavior. The
problem was not the update flag combination.

The original status-and-icon notification was restored.

### Custom Language-Bar Item GUID

The project defines `Global::JyutpingGuidLangBarInputMethodMode`, while the
button historically uses Windows' `GUID_LBI_INPUTMODE`.

Registering the button with the custom Jyutping GUID caused the Input Mode
button to disappear entirely from the Windows 11 compact input indicator.
This demonstrated that the system-defined input-mode item identity is required
for this UI surface.

The final implementation keeps `GUID_LBI_INPUTMODE`.

### Shell Icon Cache

The black-to-white change initially resembled Windows retaining one glyph and
only applying a state color. Runtime logging disproved this explanation:

- the compartment callback successfully called `OnUpdate`;
- Windows called `CLangBarItemButton::GetIcon()` after every toggle;
- `LoadImage()` succeeded each time.

The shell was refreshing exactly as requested. Jyutping was giving it the
wrong numeric resource ID.

## Icon Root Cause

Temporary high-signal logging recorded the actual icon decisions in the
deployed process:

```text
isOn=1 resourceId=21
isOn=0 resourceId=22
```

For the current resource table, ABC in a light theme must use ID `23`, not
`22`. ID `22` is `IDI_INPUT_MODE_CANTONESE_ALT`, which exactly explains the
observed white Cantonese glyph.

The project had previously normalized `resource.h` from this older ordering:

```text
21 Cantonese
22 ABC
23 Cantonese Alt
24 ABC Alt
```

to this current ordering:

```text
21 Cantonese
22 Cantonese Alt
23 ABC
24 ABC Alt
```

`Globals.obj` still predated the normalized `Define.h` and `resource.h`. It had
embedded the old integer constants:

```text
InputMethodModeCantoneseIcoIndex    = 21
InputMethodModeABCIcoIndex          = 22
InputMethodModeCantoneseAltIcoIndex = 23
InputMethodModeABCAltIcoIndex       = 24
```

Meanwhile, the newly compiled resource file used the new table. Linking the
stale object with the current `.res` produced a valid DLL with internally
inconsistent meanings for IDs `22` and `23`.

This was why source inspection, file hashes, ICO decoding, and DLL resource
inspection all looked correct while runtime behavior remained wrong.

## Resource-ID Fix

The six exported integer copies were removed from `Globals.h` and
`Globals.cpp`:

- `TextServiceIcoIndex`;
- `TextServiceAltIcoIndex`;
- `InputMethodModeCantoneseIcoIndex`;
- `InputMethodModeCantoneseAltIcoIndex`;
- `InputMethodModeABCIcoIndex`;
- `InputMethodModeABCAltIcoIndex`.

The consumers now use the current resource macros directly:

- `CompositionProcessorEngine.cpp` uses
  `INPUT_MODE_CANTONESE_ICON_INDEX`,
  `INPUT_MODE_ABC_ICON_INDEX`,
  `INPUT_MODE_CANTONESE_ALT_ICON_INDEX`, and
  `INPUT_MODE_ABC_ALT_ICON_INDEX`.
- `Register.cpp` uses `TEXTSERVICE_ICON_INDEX` and
  `TEXTSERVICE_ALT_ICON_INDEX`.

This removes an unnecessary object-file boundary between resource definitions
and their consumers. It also makes the symbolic mapping visible at the call
site where argument ordering matters.

After the source change, a full rebuild was required:

```powershell
msbuild Jyutping.sln /t:Rebuild /p:Configuration=Debug /p:Platform=ARM64 /m:1
```

The rebuild regenerated `Globals.obj`, `CompositionProcessorEngine.obj`,
`Register.obj`, `Jyutping.res`, and the final DLL from the same source state.

## Maintenance Rules

For future changes in this area:

- Do not change the Input Mode button from `GUID_LBI_INPUTMODE` without first
  verifying that it remains visible in the Windows 11 compact input indicator.
- Do not edit client text or terminate a composition directly from
  `OnPreservedKey`; request a read/write edit session and use its edit cookie.
- Preserve the operation order: commit raw text, terminate composition, then
  change Input Mode.
- Reuse `_HandleCompositionFinalizeRaw()` instead of duplicating text-store
  insertion and cleanup logic.
- Keep the live mode in `GUID_COMPARTMENT_KEYBOARD_OPENCLOSE`; do not introduce
  a parallel mode state for the language bar.
- When resource numeric IDs change, run a full rebuild for every configuration
  being shipped. A successful incremental link does not prove that every
  object embedded the current numeric values.
- When an icon glyph and its color disagree, log the selected numeric resource
  ID at runtime and compare it with the current `resource.h` table.
- Validate both sides of native resources: the compiled resource table and the
  code-side integer passed to `LoadImage()` or `RegisterProfile()`.
- Remove temporary high-volume diagnostics after the runtime evidence has
  identified the root cause.
