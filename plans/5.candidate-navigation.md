# Candidate Navigation Roadmap

This plan covers candidate-list navigation while the IME has candidates to
show. It focuses on keyboard actions for moving the highlighted candidate and
turning candidate pages. It does not change lookup ranking, candidate text
generation, candidate numbering, reverse lookup behavior, or character-standard
conversion.

## Scope

Implement these candidate actions whenever the candidate list is not empty:

- `Tab`: move to the next candidate.
- `Shift + Tab`: move to the previous candidate.
- `Arrow Up`: move to the previous candidate.
- `Arrow Down`: move to the next candidate.
- `Arrow Left`: move to the previous page.
- `Arrow Right`: move to the next page.
- `-` / `VK_OEM_MINUS`: move to the previous page.
- `=` / `VK_OEM_PLUS`: move to the next page.
- `[` / `VK_OEM_4`: move to the previous page.
- `]` / `VK_OEM_6`: move to the next page.
- `Page Up`: move to the previous page.
- `Page Down`: move to the next page.
- `Home`: jump to the first candidate of the first page.
- `End`: jump to the last candidate of the last page.

Out of scope for this roadmap:

- Making candidate page size configurable.
- Porting macOS candidate-page orientation options.
- Porting macOS bracket-key or comma/period settings.
- Adding a settings window, language-bar controls, or persistence for these
  navigation bindings.
- Changing punctuation behavior when no candidates are available.
- Changing number-key candidate selection.

## Source Of Truth

Use the draft requirements in `plans/draft.md` as the behavioral requirement.

Use the Swift/macOS implementation as a reference for page and highlight
semantics:

- `InputMethod/JyutpingInputController.swift`
- `InputMethod/InputContext.swift`
- `InputMethod/Models/PageTransformation.swift`
- `InputMethod/KeyCode/Representative.swift`

Relevant Swift behavior to mirror:

- `Tab` and `Shift + Tab` move within the displayed page.
- Moving past the displayed page boundary turns the page and highlights the
  first or last candidate of the new page.
- Page movement keeps the current highlighted position when possible.
- If the current highlighted position is beyond the new page length, page
  movement clamps to the last candidate on that page.
- Home establishes the first page and highlights the first candidate.

The Windows project should stay dedicated to the 26-key QWERTY desktop input.

## Current Windows Baseline

Relevant Windows files:

- `Jyutping/CompositionProcessorEngine.*` classifies virtual keys into
  `KEYSTROKE_FUNCTION` values and already knows the current `CANDIDATE_MODE`.
- `Jyutping/JyutpingBaseStructure.h` defines `KEYSTROKE_FUNCTION`, including
  existing move functions such as `FUNCTION_MOVE_UP`,
  `FUNCTION_MOVE_PAGE_DOWN`, `FUNCTION_MOVE_PAGE_TOP`, and
  `FUNCTION_MOVE_PAGE_BOTTOM`.
- `Jyutping/KeyStateCategory.*` dispatches candidate move functions to
  `CJyutping::_HandleCandidateArrowKey` or `CJyutping::_HandlePhraseArrowKey`.
- `Jyutping/CandidateListUIPresenter.*` translates move functions into
  `_MoveSelection`, `_MovePage`, or `_SetSelection`.
- `Jyutping/CandidateWindow.*` owns `_currentSelection`, `_candidateList`, and
  `_pageStartIndices`.
- `Jyutping/ScrollBarWindow.*` receives line, page, and position updates from
  `CCandidateWindow`.
- `Jyutping/KeyEventSink.cpp` maps `VK_PACKET` touch-keyboard page signals to
  `VK_PRIOR` and `VK_NEXT`.

Current behavior already covers some requested keys:

- `Arrow Up`, `Arrow Down`, `Page Up`, `Page Down`, `Home`, and `End` are
  classified for candidate modes.
- `_MovePage` already attempts to preserve the page-relative selection offset
  and clamp to the last candidate if the new page is shorter.
- `_SetSelection(0)` and `_SetSelection(-1)` already map to first and last
  global candidate indices.

Current gaps:

- `Tab` and `Shift + Tab` are not classified as candidate navigation keys.
- `Arrow Left` and `Arrow Right` are not candidate page keys in incremental
  candidate mode; they currently cancel/pass through so the application can move
  the caret.
- `-`, `=`, `[`, and `]` are not classified as candidate page keys.
- `AdviseUIChangedByArrowKey` ignores `FUNCTION_MOVE_LEFT` and
  `FUNCTION_MOVE_RIGHT`.
- `_MoveSelection` increments the global candidate index but does not explicitly
  resize the window when crossing a page boundary.
- Navigation methods do not expose enough small helper methods to make all
  boundary behavior obvious and testable.

## Design Goals

- Treat navigation keys as candidate actions only when candidates are present.
- Keep punctuation keys as punctuation when no candidate list is active.
- Keep all selected-candidate commit paths unchanged.
- Keep selection state centralized in `CCandidateWindow`.
- Keep TSF/UIElement notifications in `CCandidateListUIPresenter`.
- Preserve current page-relative selection on page movement when possible.
- Clamp page movement to the last candidate of the target page if the target
  page has fewer rows than the current position.
- Do not wrap from the last candidate to the first candidate, or from the first
  candidate to the last candidate, unless the action explicitly moves to a
  valid adjacent page.
- Keep scroll bar position, current page, custom candidate window size, and
  UIElement update flags synchronized after every successful navigation action.

## Navigation Semantics

Use global candidate indices internally, and derive page-local positions from
`_pageStartIndices`.

Definitions:

- `pageSize`: `_pIndexRange->Count()`.
- `currentPage`: the page containing `_currentSelection`.
- `pageStart`: `_pageStartIndices[currentPage]`.
- `pageEnd`: one before the next page start, or the last candidate index.
- `pageOffset`: `_currentSelection - pageStart`.

Move to next candidate:

1. If the current candidate is not the last candidate of the current page,
   select the next global candidate.
2. If the current candidate is the last candidate of the current page and there
   is a next page, select the first candidate of the next page.
3. If the current candidate is the last candidate overall, do nothing.

Move to previous candidate:

1. If the current candidate is not the first candidate of the current page,
   select the previous global candidate.
2. If the current candidate is the first candidate of the current page and
   there is a previous page, select the last candidate of the previous page.
3. If the current candidate is the first candidate overall, do nothing.

Move to next page:

1. If there is no next page, do nothing.
2. Select the candidate at `nextPageStart + pageOffset` if it exists within the
   target page.
3. Otherwise select the last candidate of the target page.

Move to previous page:

1. If there is no previous page, do nothing.
2. Select the candidate at `previousPageStart + pageOffset` if it exists within
   the target page.
3. Otherwise select the last candidate of the target page.

Jump to first:

- Select candidate index `0`.

Jump to last:

- Select candidate index `candidateCount - 1`.

## Phase 1: Navigation Model Cleanup

1. Add small private helpers to `CCandidateWindow` for page math:
   - `CurrentPageIndex()`
   - `PageStart(UINT page)`
   - `PageEnd(UINT page)`
   - `CurrentPageOffset()`
   - `SetSelectionAndSync(UINT index, BOOL isNotify)`

2. Keep helper return types HRESULT/BOOL consistent with the surrounding file
   style. Avoid throwing exceptions or adding STL-heavy abstractions to this
   Win32 UI class.

3. Refactor existing `_MoveSelection`, `_MovePage`, and `_SetSelection` through
   these helpers without changing behavior first.

4. Preserve `_skipEmptyPageAdjustment` behavior only where it is still needed
   for existing scroll/page-index compatibility. Remove duplicate page-boundary
   checks only after verifying the scroll bar still tracks the selection.

## Phase 2: Candidate Movement Behavior

1. Update `_MoveSelection(int offSet, BOOL isNotify)` so `+1` and `-1` follow
   the explicit cross-page semantics in this roadmap.

2. Ensure moving from the last item on one page to the next page selects that
   next page's first item.

3. Ensure moving from the first item on one page to the previous page selects
   that previous page's last item.

4. After a successful movement:
   - adjust the current page for the new selection;
   - update the scroll bar position;
   - resize the candidate window if the visible page changed;
   - invalidate the candidate window or send UIElement update flags.

5. Keep failed movement a no-op and return `FALSE`.

## Phase 3: Page Movement Behavior

1. Update `_MovePage(int offSet, BOOL isNotify)` to calculate from
   `currentPage`, `pageOffset`, and the target page bounds.

2. Preserve the current page-local position if it exists on the target page.

3. Clamp to the target page's last candidate if the target page is shorter.

4. Keep existing no-op behavior at the first page and last page.

5. Keep `_SetSelection(0, TRUE)` as the implementation for Home and
   `_SetSelection(-1, TRUE)` as the implementation for End unless Phase 1
   reveals a UIElement notification gap.

## Phase 4: Key Classification

1. Extend candidate key classification in
   `CCompositionProcessorEngine::IsVirtualKeyNeed`.

2. For `CANDIDATE_INCREMENTAL`, `CANDIDATE_ORIGINAL`,
   `CANDIDATE_WITH_NEXT_COMPOSITION`, and `CANDIDATE_PHRASE`, map:
   - `VK_TAB` with no Shift to `FUNCTION_MOVE_DOWN`.
   - `VK_TAB` with Shift to `FUNCTION_MOVE_UP`.
   - `VK_UP` to `FUNCTION_MOVE_UP`.
   - `VK_DOWN` to `FUNCTION_MOVE_DOWN`.
   - `VK_LEFT` to `FUNCTION_MOVE_PAGE_UP`.
   - `VK_RIGHT` to `FUNCTION_MOVE_PAGE_DOWN`.
   - `VK_OEM_MINUS` to `FUNCTION_MOVE_PAGE_UP`.
   - `VK_OEM_PLUS` to `FUNCTION_MOVE_PAGE_DOWN`.
   - `VK_OEM_4` to `FUNCTION_MOVE_PAGE_UP`.
   - `VK_OEM_6` to `FUNCTION_MOVE_PAGE_DOWN`.
   - `VK_PRIOR` to `FUNCTION_MOVE_PAGE_UP`.
   - `VK_NEXT` to `FUNCTION_MOVE_PAGE_DOWN`.
   - `VK_HOME` to `FUNCTION_MOVE_PAGE_TOP`.
   - `VK_END` to `FUNCTION_MOVE_PAGE_BOTTOM`.

3. Restrict the symbol-key page bindings to active candidate states so these
   keys still produce normal punctuation or raw input when the candidate list is
   empty.

4. Preserve existing `VK_LEFT` and `VK_RIGHT` caret pass-through behavior when
   there is composition text but no candidate list.

5. Preserve touch-keyboard `VK_PRIOR`/`VK_NEXT` behavior from
   `KeyEventSink.cpp`.

## Phase 5: Presenter Dispatch

1. Update `CCandidateListUIPresenter::AdviseUIChangedByArrowKey` to treat:
   - `FUNCTION_MOVE_LEFT` as previous page if callers still send it.
   - `FUNCTION_MOVE_RIGHT` as next page if callers still send it.

2. Prefer classifying left/right as page-up/page-down in Phase 4, but keep this
   compatibility mapping so older call sites cannot silently no-op.

3. Make sure the presenter sets these UIElement update flags after successful
   navigation in UI-less mode:
   - candidate moves: `TF_CLUIE_SELECTION | TF_CLUIE_CURRENTPAGE` if the page
     can change;
   - page moves: `TF_CLUIE_SELECTION | TF_CLUIE_CURRENTPAGE`;
   - home/end: `TF_CLUIE_SELECTION | TF_CLUIE_CURRENTPAGE`.

4. Keep `_HandleCandidateArrowKey` and `_HandlePhraseArrowKey` as thin
   dispatchers.

## Phase 6: Verification

Manual verification matrix:

- Candidate count less than one page.
- Candidate count exactly one page.
- Candidate count one full page plus one item.
- Candidate count multiple full pages.
- Candidate count multiple pages with a short final page.

For each matrix case, verify:

- `Tab` advances by one candidate.
- `Shift + Tab` moves back by one candidate.
- `Arrow Up` and `Arrow Down` match previous/next candidate behavior.
- `Arrow Left`, `-`, `[`, and `Page Up` move to the previous page and preserve
  page-local highlight when possible.
- `Arrow Right`, `=`, `]`, and `Page Down` move to the next page and preserve
  page-local highlight when possible.
- Moving next from a page's last candidate selects the next page's first
  candidate.
- Moving previous from a page's first candidate selects the previous page's
  last candidate.
- `Home` selects the first candidate on the first page.
- `End` selects the last candidate on the last page.
- At the first/last candidate or first/last page, invalid movement is a no-op.
- Number-key selection still selects candidates by current-page number.
- Candidate finalization still commits the highlighted candidate.
- Punctuation keys still input punctuation when no candidates are present.
- Reverse lookup candidate lists follow the same navigation rules.
- UI-less candidate mode reports correct selection/current-page changes.

msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=x64

## Implementation Notes

- `VK_OEM_PLUS` is the Windows virtual key for the `=` key on the US keyboard;
  the shifted `+` variant should not need a separate binding for this roadmap.
- `VK_OEM_4` and `VK_OEM_6` are the left and right bracket keys on the US
  keyboard.
- `Global::ModifiersValue` and `Global::CheckModifiers` should be used for
  modifier checks to match the current key-classification style.
- For `Shift + Tab`, make the Shift modifier explicit. Avoid treating
  `Control + Tab`, `Alt + Tab`, or other system combinations as candidate
  navigation.
- If a future implementation introduces unit tests, the pure page math helpers
  from Phase 1 are the right seams to test without creating Win32 windows.
