Plan: Port Emoji & Symbol candidate search from the macOS/Swift implementation.

Status: Proposed (2026-07-06).


# Emoji & Symbol Candidate Search Roadmap

This plan covers porting the macOS/Swift Emoji and Symbol candidate search into
the Windows TSF C++ input method. These are the candidates produced by Swift's
`Engine.searchSymbols(for:segmentation:)` from the bundled `symbol_table`, not
the standalone iOS emoji keyboard or symbol keyboard layouts.

The goal is to make normal Cantonese input search emoji and symbol rows by
Jyutping romanization, convert stored Unicode code-point sequences into literal
UTF-16 candidate text, and insert matching emoji/symbol candidates immediately
after their related Cantonese candidate, matching the Swift dispatch behavior.

## Source Of Truth

Use the macOS implementation as the behavior reference:

- `Modules\CoreIME\Sources\CoreIME\Emoji.swift`
  - `Engine.searchSymbols(for:segmentation:)`
  - Filters the input to syllable letters, searches `symbol_table.spell`, then
    searches exact full-length segmentation origin text.
  - Converts dotted Unicode code-point strings, for example `1F604` or
    `1F408.200D.2B1B`, into displayable Unicode text.
  - Applies `emoji_skin_map` only for category `.smileysAndPeople` (`1`) and
    `.activity` (`4`).
- `Modules\CoreIME\Sources\CoreIME\Lexicon.swift`
  - `LexiconType.emoji` and `LexiconType.symbol`.
  - The emoji/symbol initializer stores:
    - `text` as the converted emoji or symbol text.
    - `romanization` as the Cantonese romanization from `symbol_table`.
    - `input` and `mark` as the user input.
    - `attached` as the Cantonese word associated with the emoji/symbol.
  - Non-Cantonese equality and hashing use only `type` and `text`.
- `Modules\CoreIME\Sources\CoreIME\Candidate.swift`
  - `.emoji` and `.symbol` candidates display their literal text.
  - Comments are currently hidden for emoji and symbol candidates.
- `Modules\CoreIME\Sources\CoreIME\Converter.swift`
  - Normal dispatch inserts symbols after the matching Cantonese candidate:
    `item.isCantonese && item.text == symbol.attached &&
    item.romanization == symbol.romanization`.
  - Symbols that do not match a visible Cantonese candidate are not appended at
    the end.
  - Character-standard conversion applies only to Cantonese candidates.
- `Modules\Preparing\Sources\Preparing\DatabasePreparer.swift`
  - Creates `symbol_table` and `emoji_skin_map`.
  - Builds `symbol_table.spell` from the romanization after filtering to
    lowercase Basic Latin letters.
  - Creates indexes `ix_symbol_spell`, `ix_symbol_nine_key_code`, and
    `ix_emoji_skin_map_source`.
- `InputMethod\JyutpingInputController.swift`
  - Normal suggestions query symbols only when emoji suggestions are enabled.
  - Reverse lookup paths query text marks and reverse lookup candidates, but do
    not query emoji/symbol candidates.

Out of scope as source material:

- `Engine.fetchEmojiSequence()` and `Engine.fetchDefaultFrequentEmojis()`,
  because those feed the iOS emoji board and frequent-emoji UI.
- `Engine.nineKeySearchSymbols(combos:)`, because Windows remains dedicated to
  the 26-key QWERTY desktop input.
- Keyboard-side symbol panels, punctuation candidate panels, and emoji-board
  category navigation.

## Current Windows Baseline

Relevant files:

- `Jyutping\ImeTypes.*`
  - Already defines `Ime::LexiconType::Emoji` and
    `Ime::LexiconType::Symbol`.
  - Already makes non-Cantonese equality compare candidate text only after the
    candidate type matches.
  - Has `Lexicon::PlainText()`, but no emoji/symbol factory.
- `Jyutping\Resources\ime.sqlite3`
  - Already contains `symbol_table` and `emoji_skin_map`.
  - Verified schema:
    `symbol_table(id, category, unicode_version, code_point, cantonese,
    romanization, spell, nine_key_code)`.
  - Verified schema:
    `emoji_skin_map(id, source, target)`.
  - Verified current row counts:
    `symbol_table` has 7,123 rows and `emoji_skin_map` has 190 rows.
- `Jyutping\ImeDatabase.*`
  - Does not verify `symbol_table` or `emoji_skin_map`.
  - Does not expose a symbol query helper or skin-tone lookup helper.
- `Jyutping\InputEngine.*`
  - Normal `Suggest()` currently merges input memory, text marks, and Cantonese
    lexicon suggestions.
  - There is no `SearchSymbols()` API yet.
  - The current merge helper has no symbol insertion pass.
- `Jyutping\CompositionProcessorEngine.*`
  - Candidate display already converts only Cantonese candidates.
  - Non-Cantonese candidates already display literal text and empty comments.
  - Input-memory learning and forgetting already skip non-Cantonese lexicons.
  - Reverse lookup currently combines text marks with reverse lookup results;
    that should remain unchanged for this feature.
- `Jyutping\Define.h`
  - Candidate font fallback already includes `Segoe UI Emoji` and
    `Segoe UI Symbol`, which should cover most display cases.

## Scope

Implement these behaviors:

- Search bundled `symbol_table` by 26-key Cantonese romanization.
- Return emoji and symbol rows as non-Cantonese `Ime::Lexicon` values.
- Convert dotted Unicode code-point strings to proper Windows UTF-16 text,
  including surrogate pairs, variation selectors, and zero-width-joiner
  sequences.
- Apply the same default skin-tone mapping as Swift for categories `1` and `4`.
- Include emoji/symbol candidates in normal Cantonese suggestions.
- Insert emoji/symbol candidates immediately after the matching Cantonese
  candidate.
- Keep emoji/symbol candidates out of input memory and candidate forgetting.
- Display emoji/symbol candidates without Jyutping comments.
- Commit emoji/symbol candidates as literal candidate text.

Out of scope for this roadmap:

- iOS 9-key `nine_key_code` lookup.
- Emoji board UI, frequent emoji history, category browsing, or a skin-tone
  picker.
- Adding new Unicode data or regenerating the bundled SQLite database.
- Reverse lookup emoji/symbol suggestions.
- Windows text replacement integration.
- Candidate comments that show the attached Cantonese word or romanization.
- Punctuation-key symbol candidate lists.

## Desired Behavior

Normal input:

- Typing `siu` should produce normal Cantonese suggestions. Emoji rows for
  entries such as `笑 siu3` should appear immediately after the matching
  Cantonese `笑` candidate when that Cantonese candidate is present.
- Typing `sam` should allow heart emoji/symbol rows to appear after the matching
  Cantonese `心 sam1` candidate.
- Typing a multi-syllable sequence such as `singdaanloujan` should search both
  the raw syllable-letter text and exact full-length segmentation origin text.
- Rows with category `0` through `9` should become `LexiconType::Emoji`.
- Rows with category `10` or greater should become `LexiconType::Symbol`.
- Duplicate emoji/symbol rows with the same type and rendered text should be
  collapsed after preserving the earliest occurrence.

Candidate display and commit:

- Candidate text is the rendered Unicode text, not the database code-point
  string.
- Candidate comments are empty for emoji/symbol candidates.
- Character-standard conversion does not touch emoji/symbol candidate text.
- Selecting an emoji/symbol candidate commits that literal text.
- Selecting an emoji/symbol candidate does not update input memory.
- Forget-candidate commands should no-op on emoji/symbol candidates.

Reverse lookup:

- Buffers beginning with `r`, `v`, `x`, or `q` should keep the current behavior:
  text marks can be shown for the full buffer, then reverse lookup results are
  shown for the selector-specific query.
- Do not add emoji/symbol search to reverse lookup unless a separate product
  decision asks for that behavior.

## Data Model Notes

The Swift search path treats `symbol_table.code_point` as a dotted sequence of
hexadecimal Unicode scalar values. Examples:

```text
1F604
2764.FE0F
1F408.200D.2B1B
```

Windows must convert each scalar to UTF-16:

- Scalars up to `U+FFFF` become one `WCHAR`.
- Scalars above `U+FFFF` become a high-surrogate and low-surrogate pair.
- Invalid, empty, or out-of-range code-point components should skip that row.

Category handling should match Swift's search-time behavior:

- `0`: default frequent emoji rows in the source data, still emoji candidates
  when found by search.
- `1` to `8`: normal emoji categories.
- `9`: emoji-like skin-tone and related entries. Swift still treats these as
  emoji candidates because `category < 10`.
- `10`: symbol rows. These should become `LexiconType::Symbol`.

Only categories `1` and `4` should use `emoji_skin_map`:

```text
SELECT target FROM emoji_skin_map WHERE source = ?;
```

If no skin-tone target exists, use the original `code_point` string.

## Implementation Phases

### Phase 1: Database Query Support

Add symbol-table support to `ImeDatabase`.

Files:

- `Jyutping\ImeDatabase.h`
- `Jyutping\ImeDatabase.cpp`

Changes:

- Add a row type:

```cpp
struct SymbolRow
{
    int category = 0;
    int unicodeVersion = 0;
    std::wstring codePoint;
    std::wstring cantonese;
    std::wstring romanization;
};
```

- Add query helpers:

```cpp
std::vector<SymbolRow> QuerySymbolsBySpell(int64_t spell) const;
std::optional<std::wstring> QueryEmojiSkinTarget(const std::wstring& source) const;
```

- Add `VerifySchema()` checks:

```sql
SELECT category, unicode_version, code_point, cantonese, romanization
FROM symbol_table
WHERE spell = ?
LIMIT 0;

SELECT target
FROM emoji_skin_map
WHERE source = ?
LIMIT 0;
```

- Implement symbol lookup with deterministic table order:

```sql
SELECT category, unicode_version, code_point, cantonese, romanization
FROM symbol_table
WHERE spell = ?
ORDER BY rowid;
```

- Do not query or expose `nine_key_code`.

Optional refinement:

- If repeated skin-tone lookup becomes noisy, mirror `VariantLookup` with an
  `EmojiSkinLookup` statement wrapper and reuse it within one `SearchSymbols()`
  call.

### Phase 2: Code-Point Conversion And Lexicon Factories

Add the C++ equivalents of Swift's symbol conversion and symbol lexicon
initializer.

Files:

- `Jyutping\ImeTypes.h`
- `Jyutping\ImeTypes.cpp`

Recommended APIs:

```cpp
std::optional<std::wstring> SymbolTextFromCodePoints(std::wstring_view codePoints);

static Lexicon EmojiOrSymbol(
    std::wstring text,
    std::wstring cantonese,
    std::wstring romanization,
    std::wstring input,
    bool isEmoji);

bool IsEmojiOrSymbol() const;
```

Implementation notes:

- Parse dot-separated hexadecimal components.
- Reject empty components, non-hex components, surrogate scalar values, and
  values above `0x10FFFF`.
- Append valid scalars as UTF-16.
- `EmojiOrSymbol()` should set:
  - `type` to `LexiconType::Emoji` when `isEmoji` is true, otherwise
    `LexiconType::Symbol`.
  - `text` to the rendered Unicode text.
  - `romanization` to the row romanization.
  - `input` and `mark` to the user input.
  - `number` to `0`.
  - `attached` to the row Cantonese text.

### Phase 3: InputEngine Symbol Search API

Add a public symbol search API.

Files:

- `Jyutping\InputEngine.h`
- `Jyutping\InputEngine.cpp`
- Optional new file: `Jyutping\InputEngineSymbols.cpp`
- If adding a new source file, update:
  - `Jyutping\Jyutping.vcxproj`
  - `Jyutping\Jyutping.vcxproj.filters`

Recommended public API:

```cpp
std::vector<Lexicon> SearchSymbols(
    const std::vector<VirtualInputKey>& keys,
    const Segmentation& segmentation) const;
```

Optional convenience overload:

```cpp
std::vector<Lexicon> SearchSymbols(std::wstring_view input) const;
```

Port Swift's `searchSymbols(for:segmentation:)` exactly:

1. Return empty when the engine is not prepared or the key list is empty.
2. Build `syllableKeys` with `SyllableKeys(keys)`.
3. Let `syllableLength = syllableKeys.size()`.
4. Let `text = TextFromKeys(syllableKeys)`.
5. Let `input = text` when `syllableLength == keys.size()`, otherwise
   `TextFromKeys(keys)`.
6. Search regular symbols with `HashCode(text)`.
7. Filter segmentation schemes to `SchemeLength(scheme) == syllableLength`.
8. For each exact-length scheme, search `HashCode(SchemeOriginText(scheme))`.
9. Return `Distinct(regular + segmentationMatches)`.

Port Swift's `match(text:input:statement:)` behavior:

1. Query rows by spell.
2. For category `1` or `4`, replace the row `code_point` with the mapped
   `emoji_skin_map.target` when a target exists.
3. Convert the final code-point string into rendered UTF-16 text.
4. Create `Lexicon::EmojiOrSymbol(...)`.
5. Use `isEmoji = row.category < 10`.

Do not:

- Segment partial prefixes.
- Use `nine_key_code`.
- Add tone digits to the symbol spell.
- Search symbols in reverse lookup.

### Phase 4: Normal Suggestion Merge

Add symbol insertion to the existing normal suggestion dispatch.

Files:

- `Jyutping\InputEngine.cpp`

Current helper:

```cpp
MergeMemorySuggestions(memory, textMarks, queried)
```

Recommended helper shape:

```cpp
MergeMemorySuggestions(memory, textMarks, symbols, queried)
```

Preserve the current Windows ordering for memory and text marks, then insert
emoji/symbol candidates like Swift:

```cpp
std::vector<Lexicon> result;
Append(result, First(idealMemory, 3));
Append(result, textMarks);
Append(result, idealMemory);
Append(result, chained);

for (auto iterator = symbols.rbegin(); iterator != symbols.rend(); ++iterator)
{
    const Lexicon& symbol = *iterator;
    auto found = std::find_if(result.begin(), result.end(), [&symbol](const Lexicon& item)
    {
        return item.IsCantonese() &&
            symbol.attached &&
            item.text == *symbol.attached &&
            item.romanization == symbol.romanization;
    });
    if (found != result.end())
    {
        result.insert(found + 1, symbol);
    }
}

return Distinct(result);
```

Important behavioral constraints:

- Search symbols independently from Cantonese lookup, but only show a
  symbol when a matching Cantonese candidate is already in the merged list.
- Insert in reverse iteration order so final ordering matches the original
  symbol query order when multiple symbols target the same Cantonese candidate.
- Keep text marks before normal Cantonese suggestions, as currently planned and
  implemented.
- Do not add a standalone symbol section at the end of the candidate list.

### Phase 5: Settings And Runtime Gating

Swift gates this feature behind `EmojiSuggestions`, defaulting to enabled.

Recommended Windows approach:

- MVP: enable `SearchSymbols()` in normal `InputEngine::Suggest()` by default,
  matching Swift's default user experience.
- If a settings surface is available in the same implementation window, add a
  persisted `EmojiSuggestions` boolean to `ImeSettings` with default `true`.
- Store it as a `REG_DWORD` under the existing settings key:

```text
HKCU\Software\Jyutping\Settings
EmojiSuggestions    REG_DWORD    1 = on, 2 = off
```

- Gate only the merge input:

```cpp
std::vector<Lexicon> symbols = settings.emojiSuggestions
    ? SearchSymbols(keys, segmentation)
    : std::vector<Lexicon>();
```

If no user-facing settings UI exists yet, do not block the core engine port on a
registry-only toggle. Document the default-on behavior and add the setting when
the Windows settings surface can expose it.

### Phase 6: Composition And Candidate UI Review

Most TSF composition behavior should already be correct for non-Cantonese
lexicons, but verify these files:

- `Jyutping\CompositionProcessorEngine.cpp`
  - `DisplayTextForCandidate()` should continue to convert only Cantonese text.
  - `CommentTextForCandidate()` should continue to return empty text for
    non-Cantonese candidates.
  - `AppendSelectedCandidateForMemory()` and
    `CommitSelectedCandidateForMemory()` should continue to clear/skip memory
    for non-Cantonese candidates.
  - `ForgetCandidateFromMemory()` should continue to return `FALSE` for
    non-Cantonese candidates.
- `Jyutping\CandidateListUIPresenter.cpp`
  - Candidate finalization commits the displayed candidate string, so rendered
    emoji/symbol text should commit literally.
- `Jyutping\CandidateWindow.cpp`
  - Confirm DirectWrite layout measures surrogate pairs and emoji sequences
    without truncation.
  - Confirm the fallback font list reaches `Segoe UI Emoji` and
    `Segoe UI Symbol`.

No special TSF commit path should be needed if candidates are represented as
normal `CCandidateListItem` strings.

### Phase 7: Build And Verification

Build the native project:

```powershell
msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=ARM64
msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=x64
```

Database checks:

- `symbol_table` and `emoji_skin_map` are verified during `Prepare()`.
- `QuerySymbolsBySpell(HashCode(L"siu"))` returns rows such as `1F604` and
  `1F605` with `笑 siu3`.
- `QuerySymbolsBySpell(HashCode(L"sam"))` returns heart-related rows.
- Category `10` rows become `LexiconType::Symbol`.
- Invalid code-point rows, if any, are skipped without failing the whole query.

Manual IME checks:

- `siu` shows emoji candidates after the matching `笑 siu3` candidate.
- `sam` shows heart emoji/symbol candidates after the matching `心 sam1`
  candidate.
- `maau` shows cat emoji candidates after the matching `貓 maau1` candidate.
- A category `1` or `4` row that exists in `emoji_skin_map` displays the mapped
  skin-tone variant.
- Candidate comments are empty for emoji/symbol candidates.
- Simplified/Hong Kong/Taiwan character-standard options do not alter
  emoji/symbol candidate text.
- Selecting an emoji/symbol commits the rendered symbol, not the Jyutping input
  or the code-point string.
- Selecting an emoji/symbol does not create input-memory promotion on later
  input.
- Forget-candidate does not remove emoji/symbol rows.
- Reverse lookup buffers such as `rsiu`, `v...`, `x...`, and `q...` do not gain
  emoji/symbol suggestions from this feature.

## Risks And Open Questions

- Windows `wchar_t` is UTF-16, so code-point conversion must explicitly emit
  surrogate pairs. Treating code points as single `WCHAR` values will corrupt
  most emoji.
- Swift's candidate comments for `.emoji` and `.symbol` are intentionally
  disabled even though the lexicon keeps attached Cantonese text. Showing those
  comments on Windows would be a behavior change.
- The Swift search remaps only category `1` and `4` through `emoji_skin_map`.
  Applying skin-tone mapping to category `9` would be a behavior change.
- Symbol candidates are only useful when their matching Cantonese candidate is
  also visible. If Windows ranking omits the matching Cantonese candidate, the
  symbol should be omitted too, matching Swift.
- A registry-only `EmojiSuggestions` setting is hard for users to discover.
  Default-on core behavior is preferable until there is a settings UI or
  language-bar option.
- Emoji rendering depends on installed fonts and DirectWrite fallback behavior.
  The existing font list includes appropriate Windows fonts, but manual
  candidate-window inspection is still required.

## Acceptance Checklist

- `ImeDatabase::VerifySchema()` covers `symbol_table` and `emoji_skin_map`.
- `ImeDatabase` can query symbol rows by `spell`.
- Dotted Unicode code-point strings convert to valid UTF-16 candidate text.
- Emoji/symbol lexicons preserve type, rendered text, input, mark,
  romanization, and attached Cantonese text.
- `InputEngine::SearchSymbols()` matches Swift's regular and segmentation-based
  lookup behavior.
- Normal `Suggest()` searches symbols and inserts them after matching Cantonese
  candidates.
- Reverse lookup behavior remains unchanged.
- Candidate display, commit, memory, and forget behavior remain correct for
  non-Cantonese candidates.
- ARM64 and x64 builds pass.
