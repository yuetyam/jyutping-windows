# Punctuation Keys Porting Roadmap

This plan covers porting the desktop punctuation-key model and behavior from
the macOS input method into the Windows TSF input method. It covers the
physical punctuation keys on a 26-key QWERTY keyboard, their shifted forms,
the shifted number-row symbols, direct Cantonese punctuation output, and the
punctuation candidate lists used for keys that have several useful symbols.

The goal is to replace the current one-character legacy conversion table with
the Swift model: each physical key has unshifted and shifted text, an optional
instant output, and an ordered candidate list when no instant output exists.

## Implementation Result

- Added a clean `PunctuationKey` model generated directly from the Swift source,
  including hardcoded Cantonese annotations and secondary code-point text.
- Added physical-key and Shift-aware classification for OEM punctuation keys
  and shifted number-row keys.
- Added `CANDIDATE_PUNCTUATION`, which reuses the existing candidate presenter
  while bypassing `InputEngine` lexicons, input memory, and character-standard
  conversion.
- Preserved reverse-lookup apostrophes and existing unshifted normal-candidate
  navigation for minus, equal, and bracket keys.
- Reused that same navigation in punctuation candidate lists: `-` and `[` page
  backward, while `=` and `]` page forward.
- Removed the legacy punctuation table, quote toggle, and angle-bracket nesting
  state.
- Verified `Debug|ARM64` and `Debug|x64` builds with zero warnings and zero
  errors.
- Runtime verification completed by the user with the latest ARM64 build,
  including punctuation candidate paging with `-`, `=`, `[`, and `]`.

## Scope

Port these physical key groups:

- OEM punctuation keys:
  - comma / less-than
  - period / greater-than
  - slash / question mark
  - semicolon / colon
  - apostrophe / quotation mark
  - left bracket / left brace
  - right bracket / right brace
  - backslash / vertical bar
  - grave / tilde
  - minus / underscore
  - equal / plus
- Shifted number-row keys `0` through `9`.

Port these behaviors:

- English punctuation form inserts the physical key's normal ASCII text.
- Cantonese punctuation form inserts an instant Cantonese symbol when one is
  defined.
- Cantonese punctuation form opens an ordered punctuation candidate list when
  no instant symbol is defined.
- Punctuation output may contain multiple UTF-16 code units.
- Existing Cantonese input is finalized before punctuation output.
- Punctuation candidates display their annotation but do not participate in
  input memory or character-standard conversion.
- Apostrophe remains available inside reverse-lookup input where it is part of
  the query syntax.
- Existing IME and application shortcuts using Control or Alt continue to pass
  through or use their preserved-key path.

Out of scope:

- iOS 9-key layouts, touch-key long-press menus, or symbol keyboards.
- Emoji and `symbol_table` search.
- Regenerating the SQLite database.
- A standalone settings window.
- New persistent settings or options for bracket, comma, period, minus, or
  equal behavior while a normal Cantonese candidate list is open.
- Porting Swift's configurable bracket-key and comma/period-key actions. Keep
  the existing Windows candidate-list navigation behavior unchanged.
- Locale-specific non-QWERTY punctuation layouts.
- Smart punctuation based on surrounding document text.

## Target Behavior

### Direct Output

In Cantonese punctuation form, preserve Swift's instant outputs:

| Key | Output |
| --- | --- |
| `,` | `，` |
| `.` | `。` |
| `Shift+/` | `？` |
| `;` | `；` |
| `Shift+;` | `：` |
| `[` | `「` |
| `]` | `」` |
| `\` | `、` |
| `-` | `-` |
| `Shift+-` | `——` |
| `=` | `=` |
| `Shift+=` | `+` |
| `Shift+1` | `！` |
| `Shift+9` | `（` |
| `Shift+0` | `）` |

In English punctuation form, insert the physical key text, including the Shift
variant, without opening a punctuation list.

For parity with Swift, `PunctuationForm` decides OEM and shifted number-row
punctuation output. `CharacterForm` continues to control ordinary unshifted
digits and spaces, but must not apply a second full-width conversion after a
punctuation-key result has been selected.

### Candidate Lists

Open the Swift-ordered candidate list for these Cantonese inputs:

- `Shift+,` and `Shift+.`.
- `/`.
- apostrophe and quotation mark.
- `Shift+[` and `Shift+]`.
- `Shift+\`.
- grave and tilde.
- `Shift+2` through `Shift+8`.

The complete symbol values and order remain in `PunctuationKey.swift` and the
new Windows punctuation model. Do not duplicate those lists in key-handler
switch statements.

Candidate-list interaction should be:

- show the physical key text as the temporary marked composition;
- show the ordered symbols in the normal candidate window;
- number, arrow, Tab, Page Up, Page Down, Home, and End use the existing
  candidate controls;
- Enter or Space commits the highlighted punctuation candidate;
- Escape or Backspace closes the punctuation list without inserting a symbol;
- selecting a candidate commits its literal text, including multi-character
  strings;
- committing or canceling fully clears the temporary punctuation composition.

### Existing Composition

When a punctuation key is pressed during normal Cantonese composition:

1. If a candidate is highlighted, commit that candidate and apply normal input
   memory behavior.
2. Otherwise commit the raw input buffer.
3. End the normal Cantonese candidate list and composition.
4. Resolve the punctuation form.
5. Insert an instant symbol, insert English key text, or start a fresh temporary
   punctuation composition and candidate list.

This order prevents punctuation from replacing or becoming part of the
preceding Jyutping composition.

## Design Decisions

### Key Identity

Look up punctuation by `UINT` virtual-key identity and Shift state, not by the
single `WCHAR` produced by `ConvertVKey()`. The same OEM key owns both shifted
and unshifted behavior, and several outputs contain more than one `WCHAR`.

Handle only no-modifier and Shift-only punctuation events. Control, Alt, and
Windows-key combinations must remain available to the host unless an existing
preserved key explicitly owns them.

### Data Ownership

Keep punctuation data in `PunctuationKey.*`, independent of TSF composition
objects. The model should answer:

- physical key text for the Shift state;
- instant output for the Shift state;
- ordered symbols for the Shift state;
- whether the key is a number-row key.

Static string literals are sufficient for the source data. The candidate-list
builder must copy or retain text for as long as `CCandidateListItem` references
it.

Hardcode `PunctuationSymbol` comments as Cantonese text in the punctuation
model. Use the Swift Cantonese wording and code-point annotations as the source;
do not add string resources or localization plumbing for these comments.

Windows currently displays one candidate comment. Where Swift provides both a
`comment` and `secondaryComment`, combine them into that one display string
rather than adding a second candidate UI column solely for punctuation.

### Candidate Mode

Add a distinct punctuation candidate state, such as
`CANDIDATE_PUNCTUATION`. Do not disguise punctuation candidates as normal
`CANDIDATE_INCREMENTAL` suggestions: their indices do not refer to
`InputEngine` lexicons, they consume no Jyutping input keys, and selection must
not call input-memory methods with a stale suggestion index.

The punctuation state may reuse `CATEGORY_CANDIDATE` and the existing
presenter, navigation, and numeric selection machinery.

### Legacy State

Swift does not alternate smart quote output or maintain nested angle-bracket
counts. Quotes and angle brackets are explicit candidate choices. Once the new
path is complete, remove `PunctuationTable`, `CPunctuationPair`,
`CPunctuationNestPair`, `SetupPunctuationPair()`, `IsPunctuation(WCHAR)`, and
`GetPunctuation(WCHAR)` rather than keeping two competing punctuation systems.

## Risks

- `WCHAR`-based legacy APIs silently truncate multi-character Swift outputs.
  Make the active contract string-based before routing `——` or `……` through it.
- OEM virtual-key behavior depends on keyboard layout. This roadmap targets the
  project's specified physical 26-key QWERTY desktop layout; do not infer key
  identity solely from the generated character.
- Reusing normal incremental candidate mode can send punctuation indices into
  input-memory code. A distinct candidate state is required.
- Candidate items reference external text buffers. Static punctuation strings
  are stable, but combined comment storage must outlive the candidate
  presenter.
- The existing punctuation check runs after normal keystroke classification.
  Classification order must be tested or OEM keys may finalize composition
  without inserting/opening their punctuation result.
- Apostrophe is both punctuation and reverse-lookup syntax. A broad punctuation
  check will regress reverse lookup.
