# Language-Bar Settings Menu Appearance Roadmap

This roadmap covers the visual treatment of the settings menu opened by
right-clicking the `Input Mode` language-bar button. The menu should follow the
Windows light/dark app theme, use the same acrylic/translucent treatment as the
candidate window, and render with a project-controlled font fallback list while
keeping the Windows system menu font size.

The existing menu commands, persistence, labels, ordering, and left-click mode
toggle are behaviorally correct and remain unchanged by this work.

## Scope

Update the working direct Win32 popup path in
`CLangBarItemButton::ShowSettingsMenu` so it provides:

- light and dark colors selected from the Windows `AppsUseLightTheme` setting;
- CandidateWindow-style acrylic/translucent composition and rounded corners
  where the platform supports them;
- a DirectWrite font-family fallback list based on the candidate font list;
- the system default menu font size, including the user's DPI/text scaling;
- the same root menu, submenus, radio checks, disabled item, localization, and
  command IDs that exist today.

Keep `CLangBarItemButton::InitMenu` as the TSF-hosted compatibility path. Its
items must be generated from the same semantic menu description, but its final
colors, acrylic effect, and font are controlled by the TSF host and therefore
cannot be guaranteed to match the custom popup.

Out of scope:

- changing the persisted settings schema or values;
- adding a user-facing font picker or a new font-size setting;
- changing candidate-window font sizes or candidate rendering behavior;
- changing menu wording, menu order, or preserved-key shortcuts;
- replacing the `More Settings...` placeholder with a settings page;
- changing the existing left-click `InputMethodMode` toggle;
- adding a separate installer or packaging behavior.

## Current Baseline

Relevant existing code:

- `Jyutping/LanguageBar.cpp`
  - `ShowSettingsMenu` constructs several `HMENU` objects with
    `CreatePopupMenu`, `AppendMenuW`, `CheckMenuRadioItem`, and
    `TrackPopupMenu`.
  - `InitMenu` builds a second, duplicated `ITfMenu` representation.
  - `OnMenuSelect` already maps the stable command IDs to the shared settings
    handler.
- `Jyutping/LanguageBar.h`
  - `ILangBarItemButtonSettingsMenuHandler` exposes the current values and
    setters needed to build the menu.
- `Jyutping/CandidateWindow.cpp`
  - owns the current `SetWindowCompositionAttribute` acrylic implementation,
    theme-colored tint, rounded-corner setup, DirectWrite role-format helper,
    and `WM_SETTINGCHANGE` refresh behavior.
  - uses a transparent Direct2D clear so the acrylic backdrop remains visible.
- `Jyutping/Globals.cpp` and `Jyutping/Globals.h`
  - expose `UpdateSystemTheme`, `GetSystemTheme`, candidate colors, and the
    candidate/number/comment DirectWrite fallback resources.
- `Jyutping/Define.h`
  - defines `CANDIDATE_FONT_NAMES`.
- `Jyutping/Globals.cpp`
  - registers the existing `CandidateWindow`, shadow, and scrollbar window
    classes and creates the DirectWrite fallback objects.

The direct Win32 popup is the runtime path that has been verified for the
language-bar right-click interaction. Do not remove it in favor of an
`ITfMenu`-only implementation.

## Design Decision: Use a Custom Popup for the Direct Path

The system-owned `HMENU` created by `TrackPopupMenu` cannot reliably satisfy
all three visual requirements at once:

- owner-draw can customize item text, but the menu window is still owned by the
  shell/user32 menu implementation;
- the popup HWND is not available as a stable API surface for applying the
  CandidateWindow acrylic effect to every submenu;
- a menu font set through owner-draw would require an owner-window draw/measure
  contract that the current language-bar button does not have.

Add a small custom popup window for the direct right-click path. Keep the
existing `HMENU` implementation as a failure fallback so a rendering or
composition failure never removes access to settings.

The custom popup should preserve native-menu interaction semantics: cascading
submenus, hover tracking, radio checks, disabled items, keyboard navigation,
Escape/light-dismiss cancellation, and command dispatch through
`OnMenuSelect`. It should be a non-document/tool window and must not leave the
client application's focus or foreground window changed after dismissal.

## Phase 1: Create a Shared Menu Description

1. Add a small internal menu model in `LanguageBar` or a focused companion
   module. A menu item needs:
   - stable command ID (when it is a command);
   - localized display text;
   - command, submenu, or separator kind;
   - enabled/disabled state;
   - radio-checked state;
   - child items for submenus.
2. Move the current menu ordering and state calculation into one builder that
   reads `ILangBarItemButtonSettingsMenuHandler`.
3. Preserve the current ID ranges and mappings for candidate font sizes, number
   of candidates, punctuation form, character form, character variant, the
   separator, and `More Settings...`.
4. Make `ShowSettingsMenu` render the model in the custom popup.
5. Make `InitMenu` render the same model into `ITfMenu`, retaining the existing
   TSF flags (`TF_LBMENUF_SUBMENU`, radio checks, separator, and grayed item).
6. Keep `OnMenuSelect` as the single command-dispatch path. The custom window
   must not write registry settings or bypass the existing handler methods.

This phase prevents future menu additions or reordering from making the custom
popup and TSF-hosted menu disagree.

## Phase 2: Share CandidateWindow Composition Helpers

1. Extract the generic pieces currently private to `CandidateWindow.cpp` into a
   small shared module, for example `WindowAppearance.h/.cpp`:
   - dynamic lookup and invocation of `SetWindowCompositionAttribute`;
   - conversion of a `COLORREF` plus alpha to the accent gradient value;
   - DWM rounded-corner preference setup;
   - the Windows 11 capability check.
2. Keep the helper parameterized by tint color and alpha. The candidate window
   should continue passing `Global::GetCandidateWindowBackgroundColor()` and
   its current acrylic alpha so this refactor does not change its appearance.
3. Add a clear success/failure result. The settings popup must fall back to an
   opaque, theme-colored background if acrylic is unavailable, composition is
   disabled, or the API lookup fails.
4. Update the candidate window to call the shared helper and preserve its
   existing transparent Direct2D clear and `WM_SETTINGCHANGE` refresh.
5. Add `WM_SETTINGCHANGE`, `WM_THEMECHANGED`, and
   `WM_DWMCOLORIZATIONCOLORCHANGED` handling to the settings popup. Refresh
   `Global::UpdateSystemTheme`, colors, the acrylic tint, and the render target
   when one of these messages arrives.

The undocumented acrylic API remains an implementation detail shared with the
candidate window. Its existing caveat and mixed light/dark readability tests
apply to the settings popup as well.

## Phase 3: Define Menu Font Fallback and System Sizing

1. Add a settings-menu font list constant in `Jyutping/Define.h`. Start with
   the same ordered family list as `CANDIDATE_FONT_NAMES`, including CJK and
   symbol coverage. Keep the menu list named separately so a future menu-only
   adjustment cannot silently change candidate rendering.
2. Add the corresponding global font-name array/count and DirectWrite fallback
   object in `Globals.cpp`/`Globals.h`, using the existing fallback-builder
   pattern. Release it from `UninitDirectWrite` with the other fallback objects.
3. Reuse the candidate `CreateRoleTextFormat` behavior or extract that helper
   into a shared DirectWrite utility. The menu format should use the first
   configured family as its base family and the complete fallback list for all
   localized labels.
4. Read the Windows menu font metrics with
   `SystemParametersInfoForDpi(SPI_GETNONCLIENTMETRICS, ...)` (or the existing
   system fallback when the DPI-specific API is unavailable). Use
   `NONCLIENTMETRICS.lfMenuFont` to derive the DirectWrite size, preserving the
   system's default menu size and text scaling. Do not reuse
   `CandidateFontSize`, `CandidateNumberFontSize`, or any fixed menu-size
   constant.
5. Convert the logical font height to the popup monitor's DPI units before
   creating the DirectWrite format. Recreate the format and remeasure rows on
   `WM_DPICHANGED`.
6. Derive row height, horizontal padding, submenu-arrow space, and overall
   popup dimensions from the measured menu text and DPI-scaled system metrics.
   A font fallback failure must degrade to the system/default DirectWrite font,
   not prevent the menu from opening.

## Phase 4: Implement the Custom Settings Popup

1. Add `SettingsMenuWindow.h/.cpp`, deriving from `CBaseWindow` or following
   the existing CandidateWindow lifecycle. Register a dedicated window class
   and atom from `Globals.cpp`/`Globals.h`.
2. Add the new source/header files to both `Jyutping.vcxproj` and
   `Jyutping.vcxproj.filters`.
3. Create the root and cascading submenu windows with popup/tool-window
   styles. Match the candidate window's topmost behavior and use
   `WM_MOUSEACTIVATE`/focus handling so opening the menu does not activate a
   document or leave focus trapped after it closes.
4. Render with Direct2D and DirectWrite:
   - transparent clear over the acrylic backdrop when composition succeeds;
   - light/dark background and text colors from `Global`;
   - accent highlight and readable highlighted text using the same candidate
     color source;
   - separators, radio/check indicators, submenu arrows, disabled text, and
     hover state;
   - color-font drawing enabled where the selected fallback font supports it.
5. Position the root menu at `PopupPointFromClick` and cascade submenus to the
   side that has room. Clamp each popup to the monitor work area, including
   high-DPI and multi-monitor cases.
6. Implement interaction and dismissal:
   - mouse hover selects an enabled item and opens/closes submenus;
   - click invokes `OnMenuSelect` for commands and closes the menu tree;
   - disabled items and separators cannot invoke commands;
   - outside click, Escape, focus loss, and owner destruction dismiss safely;
   - arrow keys, Home/End, Up/Down, Left/Right, Enter, and mnemonic/number
     activation preserve the useful keyboard behavior of the previous native
     popup while the custom modal loop is active.
7. Restore the previous foreground/focus state on every exit path. Ensure
   `DestroyWindow` and any capture release happen on cancellation, selection,
   allocation failure, and thread teardown.

## Phase 5: Integrate and Preserve the Native Fallback

1. Update `CLangBarItemButton::ShowSettingsMenu` to:
   - build one menu-model snapshot;
   - attempt to show the custom popup;
   - route the selected ID through `OnMenuSelect`;
   - fall back to the current `HMENU`/`TrackPopupMenu` path if the custom
     window cannot be created or rendered.
2. Keep `OnClick(TF_LBI_CLK_RIGHT, ...)` on the direct popup path. Do not make
   the feature depend on the shell surfacing `ITfMenu`.
3. Update `InitMenu` to consume the shared model, preserving host-owned native
   appearance as a compatibility path.
4. Keep menu selection application immediate and use the existing settings
   handler persistence and candidate refresh behavior.
5. Verify that menu destruction does not release or invalidate the settings
   handler while `CLangBarItemButton` is still registered with TSF.

## Phase 6: Resources and Accessibility Text

No new visible labels should be needed: reuse the existing localized resource
IDs and fallback strings from `LanguageBar.cpp`. If the custom renderer needs
accessibility names or role text, derive them from the same localized labels
and menu model rather than adding duplicate strings.

Check all supported resource tables if any new accessibility/resource IDs are
introduced:

- `Jyutping/Resource.h`;
- `Jyutping/Resources/Strings.en-US.rc2`;
- `Jyutping/Resources/Strings.zh-HK.rc2`;
- `Jyutping/Resources/Strings.zh-TW.rc2`;
- `Jyutping/Resources/Strings.zh-CN.rc2`.

## Phase 7: Verification

Build serially to avoid the repository's known PDB contention:

```powershell
msbuild Jyutping.sln /m:1 /p:Configuration=Debug /p:Platform=ARM64
msbuild Jyutping.sln /m:1 /p:Configuration=Debug /p:Platform=x64
msbuild Jyutping.sln /m:1 /p:Configuration=Release /p:Platform=ARM64
```

Run `git diff --check` and confirm that the new source/header files appear in
both project files.

Manual checks:

- Light Windows theme: background, text, disabled text, radio checks, and
  highlighted rows are readable.
- Dark Windows theme: the same checks with a dark acrylic tint.
- Toggle Windows transparency effects and verify acrylic degrades to the
  themed opaque fallback without a blank or unreadable menu.
- Change the Windows theme while the menu is open and after reopening it;
  colors and tint must refresh.
- Open the menu on 100%, 150%, and 200% DPI displays. The menu font size must
  follow the system menu font metrics, not candidate font-size settings.
- Exercise CJK, Latin, tone numbers, punctuation, and symbol labels with the
  configured fallback list, including a machine where the first family is not
  installed.
- Open root and every submenu near each screen edge; verify work-area clamping.
- Verify hover, radio checks, disabled `More Settings...`, keyboard navigation,
  Escape, outside-click dismissal, and command selection.
- Confirm selecting each existing setting still updates its persisted value and
  refreshes visible candidate behavior exactly as before.
- Confirm left-click still toggles Cantonese/ABC and right-click never toggles
  the input mode.
- Exercise the TSF-hosted `InitMenu` path and confirm its item IDs, order,
  labels, and checked state match the custom popup even though the host owns
  its visual styling.
- Confirm candidate-window translucency, colors, fonts, and sizing have not
  regressed after composition/DirectWrite helper extraction.

## Risks and Mitigations

- The acrylic API is undocumented. Keep the existing dynamic lookup, use an
  opaque themed fallback, and test mixed light/dark client surfaces.
- A custom popup can regress keyboard or focus behavior. Implement the modal
  input loop and focus restoration before removing the native path, and retain
  `TrackPopupMenu` as an operational fallback.
- The TSF host controls `ITfMenu` rendering. Treat semantic parity as the
  contract for that path; do not attempt process-global undocumented menu-theme
  hooks to force its appearance.
- New window classes and DirectWrite resources add teardown paths. Follow the
  existing `CBaseWindow`/`CandidateWindow` ownership pattern and test
  allocation failure, DLL detach, and owner-window destruction.

## Definition of Done

The roadmap is complete when the direct language-bar settings popup is a
usable custom-rendered window with Windows light/dark colors, visible
CandidateWindow-style translucency (or a readable themed fallback), the
configured font fallback list, and system-default font sizing; all existing
settings behavior remains intact; the TSF menu path remains semantically in
sync; the listed ARM64/x64 builds pass; and the manual theme, DPI, keyboard,
focus, localization, fallback-font, and regression checks have been verified.
