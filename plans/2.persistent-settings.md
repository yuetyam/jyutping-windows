Plan: Implement persistent settings for IME configuration.

Status: Implemented (2026-07-03).


# Persistent Settings Roadmap

This plan covers adding a persistent settings layer to the Windows TSF C++
project. The first setting to implement is `InputMethodMode`, which controls
whether the IME starts in Cantonese mode or ABC mode and is toggled by pressing
the `Shift` key once.

The implementation should be designed as a small settings system, not as a
one-off registry write for the IME mode. Future settings such as punctuation,
character width, candidate UI options, and reverse lookup preferences should be
able to reuse the same storage, defaults, validation, and synchronization path.

## Current Baseline

Relevant files:

- `Jyutping/CompositionProcessorEngine.*`: owns preserved keys, language bar
  setup, TSF compartments, and synchronization with conversion mode.
- `Jyutping/KeyEventSink.cpp`: reads the current open/closed, width, and
  punctuation compartments while deciding whether to eat key events.
- `Jyutping/Compartment.*`: wraps TSF compartment get/set operations.
- `Jyutping/LanguageBar.*`: binds language bar buttons to compartments.
- `Jyutping/RegKey.*`: existing registry helper that can be reused for
  per-user settings persistence.
- `Jyutping/Globals.*`: owns GUIDs and process-wide constants.

The current `InputMethodMode` behavior is compartment-driven:

- Cantonese mode: `GUID_COMPARTMENT_KEYBOARD_OPENCLOSE = TRUE`
- ABC mode: `GUID_COMPARTMENT_KEYBOARD_OPENCLOSE = FALSE`
- `KeyboardOpenCompartmentUpdated()` mirrors open/closed state into
  `TF_CONVERSIONMODE_NATIVE`.
- `ConversionModeCompartmentUpdated()` mirrors external conversion-mode changes
  back into the open/closed compartment.
- `OnPreservedKey()` handles the `Shift` preserved key and toggles
  `GUID_COMPARTMENT_KEYBOARD_OPENCLOSE`.
- `InitializeJyutpingCompartment()` currently resets the IME to Cantonese mode
  every time by setting the open/closed compartment to `TRUE`.
- `SetupLanguageProfile()` currently calls `InitializeJyutpingCompartment()`
  before `SetupLanguageBar()`, while `_pCompartmentConversion` is created inside
  `SetupLanguageBar()`. Therefore the existing
  `PrivateCompartmentsUpdated(pThreadMgr)` call at the end of
  `InitializeJyutpingCompartment()` returns early and does not synchronize
  `GUID_COMPARTMENT_KEYBOARD_INPUTMODE_CONVERSION` during startup.

## Design Goals

- Persist user settings per Windows user, not per process activation.
- Keep TSF compartments as the live runtime state; use persistent settings as
  the source for startup defaults and as the destination for user-initiated
  changes.
- Implement `InputMethodMode` first without blocking future settings.
- Avoid persistence feedback loops when compartment callbacks mirror TSF state.
- Validate stored values and fall back to defaults if values are missing or
  invalid.
- Keep writes small and synchronous for simple toggles; avoid background
  infrastructure until there is a real need.
- Keep registration and COM activation independent from user settings. The IME
  should run with defaults if settings cannot be read or written.

## Proposed Settings Model

Add a small settings type, for example `Settings.h` and `Settings.cpp`.

Core types:

```cpp
enum class InputMethodMode : DWORD
{
    Cantonese = 1,
    ABC = 2
};

struct ImeSettings
{
    DWORD version = 1;
    InputMethodMode inputMethodMode = InputMethodMode::Cantonese;
};
```

Recommended API shape:

```cpp
class SettingsStore
{
public:
    ImeSettings Load() const;
    bool Save(const ImeSettings& settings) const;
    bool SaveInputMethodMode(InputMethodMode mode) const;

private:
    bool ReadDWORD(_In_z_ PCWSTR valueName, _Out_ DWORD& value) const;
    bool WriteDWORD(_In_z_ PCWSTR valueName, DWORD value) const;
};
```

Use explicit enum values and `DWORD` registry storage so values are stable and
easy to inspect. Keep conversion helpers close to the settings model:

```cpp
InputMethodMode InputMethodModeFromKeyboardOpen(BOOL isOpen);
BOOL KeyboardOpenFromInputMethodMode(InputMethodMode mode);
```

## Storage Location

Use a per-user registry key. Suggested location:

```text
HKCU\Software\Jyutping\Settings
```

Suggested values:

```text
Version           REG_DWORD  1
InputMethodMode   REG_DWORD  1 = Cantonese, 2 = ABC
CharacterForm     REG_DWORD  1 = HalfWidth, 2 = FullWidth
PunctuationForm   REG_DWORD  1 = Cantonese, 2 = English
```

Rationale:

- `HKCU` matches per-user IME preferences and does not require elevation.
- `CRegKey` already supports `REG_DWORD` read/write operations.
- A top-level `Settings` key leaves room for future grouped values without
  coupling settings to COM registration keys.

## Phase 1: Add Settings Infrastructure

1. Add `Settings.h` and `Settings.cpp`.

2. Add the files to:
   - `Jyutping/Jyutping.vcxproj`
   - `Jyutping/Jyutping.vcxproj.filters`

3. Implement defaults in one place:
   - `InputMethodMode::Cantonese`
   - future defaults should be added to `ImeSettings`, not scattered through
     compartment initialization code.

4. Implement `SettingsStore::Load()`:
   - Open `HKCU\Software\Jyutping\Settings` with `KEY_READ`.
   - If the key or value is missing, return defaults.
   - If `InputMethodMode` is outside the known enum range, return the default
     for that field.
   - Read `Version`, but do not require migration logic while version is `1`.

5. Implement `SettingsStore::Save()`:
   - Create `HKCU\Software\Jyutping\Settings` with `KEY_READ | KEY_WRITE`.
   - Write `Version`.
   - Write all known settings fields.
   - Return `false` on registry failure, but do not make this fatal to IME
     activation.

6. Implement `SettingsStore::SaveInputMethodMode()` as a focused write helper
   used by the Shift toggle path. It can load current settings first, update one
   field, then save the full struct, or write the single value if the schema
   remains simple.

## Phase 2: Apply Persisted InputMethodMode On Activation

1. Add a settings member to `CCompositionProcessorEngine`, for example:

   ```cpp
   ImeSettings _settings;
   SettingsStore _settingsStore;
   BOOL _isApplyingSettings;
   BOOL _isMirroringConversionMode;
   ```

2. Initialize the new guard flags to `FALSE` in
   `CCompositionProcessorEngine::CCompositionProcessorEngine()`. This codebase
   initializes members explicitly in the constructor, so do not rely on default
   member initialization for the runtime guards.

3. Load settings during `SetupLanguageProfile()` before the initial compartment
   values are applied.

4. Replace the hard-coded initial mode in `InitializeJyutpingCompartment()`:

   ```cpp
   CompartmentKeyboardOpen._SetCompartmentBOOL(
       KeyboardOpenFromInputMethodMode(_settings.inputMethodMode));
   ```

5. Keep current defaults for the existing non-persistent settings at first:
   - `CharacterForm = HalfWidth`
   - `PunctuationForm = Cantonese`

6. Do not rely on `PrivateCompartmentsUpdated(pThreadMgr)` inside
   `InitializeJyutpingCompartment()` with the current call order. At that point
   `_pCompartmentConversion` has not been created yet, so the sync returns
   early.

7. Use one of these startup synchronization approaches:
   - Preferred minimal change: keep `InitializeJyutpingCompartment()` before
     `SetupLanguageBar()`, but move the initial conversion-mode sync to
     `SetupLanguageProfile()` immediately after `SetupLanguageBar()` returns.
     At that point `_pCompartmentConversion` exists, so
     `PrivateCompartmentsUpdated(pThreadMgr)` can mirror the loaded open/closed,
     width, and punctuation compartments into
     `GUID_COMPARTMENT_KEYBOARD_INPUTMODE_CONVERSION`.
   - Broader cleanup: split conversion-compartment creation out of
     `SetupLanguageBar()` into a dedicated setup helper that runs before
     `InitializeJyutpingCompartment()`, then keep the initial sync at the end of
     initialization.

8. The preferred startup order should be:

   ```text
   SetupPreserved()
   _settings = _settingsStore.Load()
   _isApplyingSettings = TRUE
   InitializeJyutpingCompartment()
   SetupPunctuationPair()
   SetupLanguageBar()
   PrivateCompartmentsUpdated()
   _isApplyingSettings = FALSE
   SetupKeystroke()
   SetupConfiguration()
   SetupInputEngine()
   ```

9. Ensure language bar icons are initialized after compartments reflect the
   loaded setting. If existing language bar registration already listens to
   compartment values, no special language bar code should be needed.

## Phase 3: Persist Shift Toggles

1. Update the `Shift` preserved-key branch in
   `CCompositionProcessorEngine::OnPreservedKey()`.

2. Preserve the current toggle order:
   - Read `GUID_COMPARTMENT_KEYBOARD_OPENCLOSE`.
   - Write the opposite value.
   - Mark the key as eaten.

3. Check the `HRESULT` from both compartment operations. The current code
   ignores `_GetCompartmentBOOL()` and `_SetCompartmentBOOL()` results; the
   persistence implementation must not save a mode that failed to become the
   live runtime state.

   ```cpp
   BOOL isOpen = FALSE;
   HRESULT hr = CompartmentKeyboardOpen._GetCompartmentBOOL(isOpen);
   if (FAILED(hr))
   {
       *pIsEaten = FALSE;
       return;
   }

   BOOL newIsOpen = isOpen ? FALSE : TRUE;
   hr = CompartmentKeyboardOpen._SetCompartmentBOOL(newIsOpen);
   if (FAILED(hr))
   {
       *pIsEaten = FALSE;
       return;
   }
   ```

4. After a successful compartment write, update in-memory settings and save the
   new mode:

   ```cpp
   InputMethodMode mode = InputMethodModeFromKeyboardOpen(newIsOpen);
   _settings.inputMethodMode = mode;
   _settingsStore.SaveInputMethodMode(mode);
   ```

5. Do not block the toggle if the registry write fails. The runtime mode should
   still change; persistence failure should only mean the next activation falls
   back to the last successfully saved value or the default.

6. Consider logging registry write failures through the existing logger if the
   settings layer can do so without creating unwanted dependencies.

## Phase 4: Synchronize External Mode Changes

Mode can change outside the preserved-key path:

- Language bar button toggles the open/closed compartment.
- Windows or TSF may update `GUID_COMPARTMENT_KEYBOARD_INPUTMODE_CONVERSION`.
- Profile activation may reapply compartments.

Add a clear policy:

- Persist deliberate user preference changes after initialization has completed:
  the `Shift` preserved key and language bar open/closed toggles.
- Do not persist open/closed changes that only mirror
  `GUID_COMPARTMENT_KEYBOARD_INPUTMODE_CONVERSION`. Those TSF/host conversion
  mode changes are runtime state synchronization, not a saved startup
  preference.
- Do not persist while applying settings during activation.
- Avoid duplicate writes when `KeyboardOpenCompartmentUpdated()` and
  `ConversionModeCompartmentUpdated()` bounce values between compartments.

Suggested implementation:

1. Add `_isApplyingSettings = TRUE` while initial settings are being written to
   compartments.

2. Add a second guard, for example `_isMirroringConversionMode`, around
   `ConversionModeCompartmentUpdated()` when it writes
   `GUID_COMPARTMENT_KEYBOARD_OPENCLOSE` from `TF_CONVERSIONMODE_NATIVE`.

3. Clear `_isApplyingSettings` after initial synchronization completes.

4. In `KeyboardOpenCompartmentUpdated()`, after conversion mode is synchronized,
   persist the current open/closed value only when both guards are false:

   ```cpp
   if (!_isApplyingSettings && !_isMirroringConversionMode)
   {
       PersistInputMethodModeFromCompartment(pThreadMgr);
   }
   ```

5. Keep the Shift branch save from Phase 3, or centralize persistence in
   guarded `KeyboardOpenCompartmentUpdated()`. If centralizing there, make sure
   the callback always fires after `OnPreservedKey()` changes the compartment.

Preferred approach:

- Centralize persistence in a helper such as
  `PersistInputMethodModeFromCompartment(pThreadMgr)`.
- Call the helper from `KeyboardOpenCompartmentUpdated()` only when
  `_isApplyingSettings == FALSE` and `_isMirroringConversionMode == FALSE`.
- Treat `ConversionModeCompartmentUpdated()` as a one-way mirror from external
  conversion-mode state into live compartments. It should not update persisted
  `InputMethodMode`.
- In the Shift branch, save directly only if testing shows the compartment sink
  does not fire reliably after `SetValue()`.

## Phase 5: Prepare Future Settings

After `InputMethodMode` works, the same infrastructure should support other
settings with minimal changes.

Likely future settings:

- `PunctuationForm`: Cantonese punctuation vs English punctuation.
- `CharacterForm`: full-width vs half-width output.
- `CandidatePageSize`: number of candidates per page.
- `CandidateFontSize`: candidate window text size.
- `DefaultReverseLookupMethod` or reverse lookup behavior flags, if needed.

Rules for adding future settings:

- Add a field to `ImeSettings`.
- Add a default value.
- Add validation while loading.
- Add registry value names in one place.
- Add compartment synchronization only for settings that have live TSF
  compartments.
- Do not make every setting a TSF compartment if it is only used internally.

## Phase 6: Tests And Manual Verification

Build checks:

```powershell
msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=x64
msbuild Jyutping.sln /p:Configuration=Release /p:Platform=x64
msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=ARM64
msbuild Jyutping.sln /p:Configuration=Release /p:Platform=ARM64
```

Manual verification:

1. Remove or rename `HKCU\Software\Jyutping\Settings`.
2. Activate the IME.
3. Confirm the first activation defaults to Cantonese mode.
4. Press `Shift` once.
5. Confirm the IME switches to ABC mode and the language bar icon updates.
6. Deactivate/reactivate the IME or restart the test app.
7. Confirm ABC mode is restored.
8. Press `Shift` again.
9. Confirm Cantonese mode is restored and persisted.
10. Toggle mode from the language bar and confirm it persists according to the
    Phase 4 policy.
11. Corrupt `InputMethodMode` with an invalid `REG_DWORD` and confirm the IME
    falls back to Cantonese mode without activation failure.

Regression checks:

- Normal Jyutping input works in Cantonese mode.
- Key events pass through in ABC mode.
- Reverse lookup still works after returning to Cantonese mode.
- `Shift+Space` still toggles character width.
- `Ctrl+.` still toggles punctuation.
- Conversion mode changes still update existing compartments.

## Implementation Notes And Risks

- The current `OnPreservedKey()` ends with `*pIsEaten = TRUE` even after the
  unknown-GUID branch sets it to `FALSE`. Do not broaden that behavior while
  implementing settings; fix it separately only if it is confirmed as a bug and
  tested.
- `InitializeJyutpingCompartment()` currently overwrites every activation with
  Cantonese mode. This is the main behavior that must change for persistence.
- `CJyutping::_Deactivate()` clears compartments. That should remain a runtime
  cleanup operation and should not delete persisted settings.
- Registry failures should be non-fatal. The IME should continue with defaults
  or current runtime state.
- Keep settings code independent from UI classes so it can be reused by future
  configuration surfaces.

## Completion Criteria

- `InputMethodMode` is loaded from per-user storage on activation.
- Pressing `Shift` toggles Cantonese/ABC mode and persists the new mode.
- The persisted mode survives IME deactivation/reactivation and process restart.
- Missing or invalid settings fall back to default values.
- The settings layer has a clear place to add future fields without modifying
  unrelated TSF code.
