Plan: Add selectable `CharacterStandard` options to the IME runtime.

Status: Planned.


# CharacterStandard Options Roadmap

This plan covers adopting the already-ported `CharacterStandard` converters in
the active Windows IME flow. It adds user-selectable character variants,
persistence, and candidate display conversion matching the macOS/Swift behavior
for the four legacy character-standard options.

The converter implementation itself was already completed in
`plans/3.character-standard-converters.md` and should not be re-ported here.

## Scope

Add runtime support for these user-facing options:

- `Control + Shift + 1`: Traditional (Preset), default and fallback.
- `Control + Shift + 2`: Traditional (HongKong).
- `Control + Shift + 3`: Traditional (Taiwan).
- `Control + Shift + 4`: Simplified, internally backed by
  `CharacterStandard::Mutilated`.

Only the number-row keys are in scope. Numpad keys should not be registered as
equivalent shortcuts for this roadmap.

Persist the selected option across sessions using the existing per-user settings
store:

```text
HKCU\Software\Jyutping\Settings

CharacterVariant  REG_DWORD  1 = Traditional
                             2 = HongKong
                             3 = Taiwan
                             4 = Simplified
```

Apply the selected variant when building display candidates:

- `Candidate.text` should be the converted form of `Lexicon.text`.
- `Candidate.comment` should remain the Cantonese romanization or reverse lookup
  annotation.
- Candidate selection should commit the converted candidate text because the
  selected `CCandidateListItem::_ItemString` is what current commit paths use.

Out of scope for this roadmap:

- Re-porting `CharacterStandard` converters.
- Porting the macOS `TraditionalCharacterStandard` option set.
- Adding shortcuts for number-row `5`, `6`, `7`, `8`, `9`, or `0`; those keys
  should remain reserved for future non-`CharacterStandard` options.
- Adding a settings window, language-bar menu, or indicator UI for the selected
  character variant.
- Changing reverse lookup trigger keys or the 26-key QWERTY input model.
- Converting raw input buffers, reading strings, Jyutping marks, or romanization
  comments.

## Source Of Truth

Use these Windows files as the implementation baseline:

- `Jyutping/CharacterStandard.*`: already contains the enum, raw-value helpers,
  and `Ime::ConvertText`.
- `Jyutping/Settings.*`: existing registry-backed settings model.
- `Jyutping/CompositionProcessorEngine.*`: owns preserved keys, live settings,
  input suggestions, and `CCandidateListItem` construction.
- `Jyutping/InputEngine.*`: owns the `ImeDatabase` instance needed by
  `Ime::ConvertText`.
- `Jyutping/Globals.*`: owns preserved-key GUIDs and user-facing descriptions.
- `Jyutping/KeyEventSink.cpp`: forwards TSF preserved-key events to
  `CCompositionProcessorEngine::OnPreservedKey`.
- `Jyutping/SearchCandidateProvider.cpp`: exposes candidates through the same
  `GetCandidateList` path and should inherit conversion automatically.

Use the macOS/Swift version only for behavior:

- `InputMethod/Options.swift`
- `InputMethod/JyutpingInputController.swift`
- `Modules/CoreIME/Sources/CoreIME/Candidate.swift`
- `Modules/CoreIME/Sources/CoreIME/Converter.swift`

Important Swift behavior to mirror:

- The legacy user-facing character-standard setting stores compact values:
  `1 = Traditional`, `2 = HongKong`, `3 = Taiwan`, `4 = Simplified`.
- Menu shortcuts use `Control + Shift + 1` through `Control + Shift + 4`.
- Display candidate text is derived from the selected charset, while the
  underlying lexicon and romanization remain intact.

Important Swift behavior to intentionally skip:

- When the selected legacy option is Traditional, macOS may route through
  `traditionalCharacterStandard`; Windows should not port that second setting in
  this roadmap. Traditional means `CharacterStandard::Preset`.

## Current Windows Baseline

The Windows project already has the reusable converter layer:

- `CharacterStandard::Preset`, `HongKong`, `Taiwan`, and `Mutilated` exist.
- `Ime::ConvertText(const ImeDatabase&, std::wstring_view, CharacterStandard)`
  performs the actual conversion.
- `CharacterStandard::Preset` is a no-op and should remain the default fallback.

The settings layer already persists:

- `InputMethodMode`
- `CharacterForm`
- `PunctuationForm`

The preserved-key layer already registers:

- `Shift`: input method mode toggle.
- `Shift + Space`: half-width/full-width toggle.
- `Control + .`: Cantonese/English punctuation toggle.

The candidate generation path currently does not apply `CharacterStandard`:

```text
CCompositionProcessorEngine::GetCandidateList
  -> AppendInputEngineCandidates
  -> GetInputSuggestions
  -> Ime::InputEngine::Suggest / ReverseLookup
  -> CCandidateListItem::_ItemString = suggestion.text
  -> CCandidateListItem::_ItemComment = suggestion.romanization
```

This roadmap should change only the candidate item text materialization, not the
lookup result ranking, input-count handling, or raw buffer state.

## Design Goals

- Keep `CharacterStandard` raw values Swift-compatible in `CharacterStandard.h`.
- Add a separate user-setting enum for the four legacy options because the
  registry values requested by this roadmap are compact values `1` through `4`,
  not the internal `CharacterStandard` raw values.
- Make missing, invalid, or unreadable settings fall back to Traditional
  (`CharacterStandard::Preset`).
- Convert display text as late as practical, when building
  `CCandidateListItem`, so cached `Ime::Lexicon` suggestions remain canonical.
- Preserve `Lexicon.inputCount`, raw input text, candidate comments, reading
  strings, and reverse lookup marks.
- Avoid exposing `ImeDatabase` directly to TSF UI code. Add a small
  `Ime::InputEngine` wrapper for conversion if needed.
- Treat registry write failures as non-fatal. The in-memory selection should
  still change for the current session after a successful shortcut action.
- Keep future number-row options possible by registering only `1` through `4`
  in this roadmap.

## Proposed Settings Model

Add a user-facing enum to `Settings.h`:

```cpp
enum class CharacterVariant : DWORD
{
    Traditional = 1,
    HongKong = 2,
    Taiwan = 3,
    Simplified = 4
};
```

Extend `ImeSettings`:

```cpp
struct ImeSettings
{
    DWORD version = 1;
    InputMethodMode inputMethodMode = InputMethodMode::Cantonese;
    CharacterForm characterForm = CharacterForm::HalfWidth;
    PunctuationForm punctuationForm = PunctuationForm::Cantonese;
    CharacterVariant characterVariant = CharacterVariant::Traditional;
};
```

Add helpers close to the settings model:

```cpp
CharacterVariant CharacterVariantFromRawValue(DWORD value);
CharacterStandard CharacterStandardFromCharacterVariant(CharacterVariant variant);
```

Mapping:

```text
CharacterVariant::Traditional -> CharacterStandard::Preset
CharacterVariant::HongKong    -> CharacterStandard::HongKong
CharacterVariant::Taiwan      -> CharacterStandard::Taiwan
CharacterVariant::Simplified  -> CharacterStandard::Mutilated
```

Add `SettingsStore::SaveCharacterVariant(CharacterVariant variant)`.

The existing `Version` value may remain compatible if the loader treats missing
`CharacterVariant` as the default. If the project wants schema visibility, bump
`CurrentSettingsVersion` and still keep loading old settings without migration
failure.

## Phase 1: Persist CharacterVariant

1. Add `CharacterVariant` to `Settings.h`.

2. Add validation in `Settings.cpp`:
   - valid values: `1`, `2`, `3`, `4`
   - invalid values fall back to `CharacterVariant::Traditional`

3. Add `CharacterVariantValueName = L"CharacterVariant"`.

4. Update `SettingsStore::Load()`:
   - read `CharacterVariant` from `HKCU\Software\Jyutping\Settings`
   - preserve all existing fallback behavior for other settings
   - default to Traditional if the value is missing or invalid

5. Update `SettingsStore::Save()`:
   - write `CharacterVariant`
   - keep existing values for input mode, width, and punctuation

6. Add `SettingsStore::SaveCharacterVariant()`.

7. Ensure `CCompositionProcessorEngine::_settings` receives the loaded
   `characterVariant` during `SetupLanguageProfile()` and
   `ApplyPersistedSettings()` through the existing `_settingsStore.Load()` path.

## Phase 2: Add CharacterVariant Preserved Keys

1. Add four preserved-key GUIDs in `Globals.h` and `Globals.cpp`.
   Suggested names:
   - `JyutpingGuidTraditionalCharacterVariantPreserveKey`
   - `JyutpingGuidHongKongCharacterVariantPreserveKey`
   - `JyutpingGuidTaiwanCharacterVariantPreserveKey`
   - `JyutpingGuidSimplifiedCharacterVariantPreserveKey`

2. Add user-facing descriptions in `Globals.cpp`.
   Suggested text:
   - `Character Variant (Traditional, Ctrl+Shift+1)`
   - `Character Variant (HongKong, Ctrl+Shift+2)`
   - `Character Variant (Taiwan, Ctrl+Shift+3)`
   - `Character Variant (Simplified, Ctrl+Shift+4)`

3. Add preserved-key members to `CCompositionProcessorEngine`.
   Use distinct GUIDs per action. Do not register all four number keys under one
   GUID, because `OnPreservedKey` receives the GUID but not the number key.

4. Register these preserved keys in `SetupPreserved()`:

   ```cpp
   TF_PRESERVEDKEY key;
   key.uVKey = L'1';
   key.uModifiers = TF_MOD_CONTROL | TF_MOD_SHIFT;
   ```

   Repeat for `L'2'`, `L'3'`, and `L'4'`.

5. Initialize the new preserved-key registrations with `InitPreservedKey()`.

6. Update `CCompositionProcessorEngine::OnPreservedKey()`:
   - map each new GUID to the corresponding `CharacterVariant`
   - update `_settings.characterVariant`
   - call `_settingsStore.SaveCharacterVariant(variant)`
   - set `*pIsEaten = TRUE` only after the action is recognized

7. While editing preserved-key dispatch, fix or avoid broadening the existing
   unconditional final `*pIsEaten = TRUE` behavior. Unknown preserved-key GUIDs
   should remain not eaten.

8. Do not register `Control + Shift + 5` through `Control + Shift + 0` in this
   roadmap. Leave those combinations available for future options.

## Phase 3: Add Conversion Access Through InputEngine

`CCompositionProcessorEngine` needs converted text, but the converter requires
`ImeDatabase`, which is currently owned by `Ime::InputEngine`.

Preferred small API:

```cpp
std::wstring InputEngine::ConvertText(
    std::wstring_view text,
    CharacterStandard standard) const;
```

Implementation:

```cpp
return Ime::ConvertText(_database, text, standard);
```

Include `CharacterStandard.h` in `InputEngine.h` or forward declare the enum if
practical.

Do not add a public `ImeDatabase&` getter just for candidate UI conversion
unless another existing engine API already follows that pattern.

## Phase 4: Convert Candidate Display Text

1. Add a helper in `CCompositionProcessorEngine`, for example:

   ```cpp
   CharacterStandard CurrentCharacterStandard() const;
   std::wstring DisplayTextForCandidate(const Ime::Lexicon& suggestion) const;
   ```

2. `CurrentCharacterStandard()` should map `_settings.characterVariant` through
   `CharacterStandardFromCharacterVariant()`.

3. In `AppendInputEngineCandidates()`, replace:

   ```cpp
   _candidateItemTextStorage.push_back(suggestion.text);
   ```

   with converted display text:

   ```cpp
   _candidateItemTextStorage.push_back(
       _inputEngine.ConvertText(suggestion.text, CurrentCharacterStandard()));
   ```

4. Keep comments unchanged:

   ```cpp
   _candidateItemCommentStorage.push_back(suggestion.romanization);
   ```

5. Preserve candidate input-count logic exactly:
   - normal suggestions use `suggestion.inputCount`
   - reverse lookup candidates use `consumedQueryLength + 1`

6. Apply conversion to all `suggestion.text` values through the same path.
   Non-Han text, symbols, and unsupported scalars should pass through unchanged
   because `Ime::ConvertText` already preserves unmapped text.

7. Do not mutate `_cachedSuggestions`. The cache key can remain input text plus
   reverse lookup method because variant conversion happens after suggestions
   are retrieved.

8. Consider a tiny fast path for `CharacterVariant::Traditional` if profiling
   shows a need. It is not required for correctness because
   `CharacterStandard::Preset` is no-op.

## Phase 5: Refresh Runtime State After Shortcut Changes

The minimum required behavior is that newly generated candidate lists use the
newly selected character variant immediately after the shortcut is handled.

Preferred behavior:

1. After a character-variant shortcut is eaten, keep the current raw input buffer
   and composition state.

2. If a candidate list is visible, rebuild candidate items from
   `GetCandidateList()` and update the presenter with the converted text.
   This likely needs a small result from
   `CCompositionProcessorEngine::OnPreservedKey()` so
   `CJyutping::OnPreservedKey()` can tell whether the action changed the
   character variant.

3. Do not clear `_cachedSuggestions` solely because the character variant
   changed. Rebuilding `CCandidateListItem` from the same suggestions is enough.

4. Do not commit buffered text, cancel composition, or change ABC/Cantonese mode
   while switching character variants.

5. If immediate visible refresh is too invasive for the first implementation,
   document the limitation and ensure the next candidate-list rebuild uses the
   selected variant. The persisted setting and subsequent sessions must still be
   correct.

## Phase 6: Commit And Search Integration Checks

1. Confirm candidate selection commits the converted text.
   Current paths read `_pCandidateListUIPresenter->_GetSelectedCandidateString`,
   which should return `CCandidateListItem::_ItemString`.

2. Confirm incremental candidate selection still computes tail input from
   `_InputCount`, not from display text length.

3. Confirm reverse lookup candidates display:
   - converted candidate text
   - unchanged Cantonese romanization comment
   - unchanged reverse lookup pre-edit mark

4. Confirm `CSearchCandidateProvider::GetSearchCandidates()` inherits converted
   candidate strings because it calls `GetCandidateList()`.

5. Confirm direct punctuation and full-width character-form output do not call
   the character-standard converter.

## Phase 7: Verification

Build checks:

```powershell
msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=x64
msbuild Jyutping.sln /p:Configuration=Release /p:Platform=x64
msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=ARM64
msbuild Jyutping.sln /p:Configuration=Release /p:Platform=ARM64
```

Manual registry checks:

1. Delete or rename `HKCU\Software\Jyutping\Settings\CharacterVariant`.
2. Activate the IME.
3. Confirm candidates use Traditional/Preset behavior.
4. Press `Control + Shift + 2`.
5. Confirm `CharacterVariant` is written as `2`.
6. Restart the test app or reactivate the IME.
7. Confirm HongKong remains selected.
8. Repeat for `Control + Shift + 1`, `3`, and `4`.
9. Set `CharacterVariant` to an invalid `REG_DWORD`, then reactivate the IME.
10. Confirm the IME falls back to Traditional/Preset without activation failure.

Candidate conversion checks:

1. Choose at least one known `variant_hk` row and verify the same Jyutping input
   displays the HongKong form under `Control + Shift + 2`.
2. Choose at least one known `variant_tw` row and verify the Taiwan form under
   `Control + Shift + 3`.
3. Verify Simplified under `Control + Shift + 4` with phrase-aware examples
   already covered by converter behavior, such as:
   - `反覆` -> `反复`
   - `乾乾淨淨` -> `干干净净`
   - `甚麼` -> `什么`
   - `項鍊` -> `项链`
4. Press `Control + Shift + 1` and confirm the same candidates return to the
   original Traditional/Preset display.
5. Verify romanization comments stay unchanged for every variant.
6. Verify selecting a converted candidate commits the converted text.
7. Verify reverse lookup candidates convert display text but keep the Cantonese
   romanization comment.

Shortcut regression checks:

- `Shift` still toggles Cantonese/ABC mode.
- `Shift + Space` still toggles half-width/full-width mode.
- `Control + .` still toggles Cantonese/English punctuation.
- Number-row candidate selection still works when `Control + Shift` is not held.
- Numpad keys do not trigger character-variant changes.
- `Control + Shift + 5` through `Control + Shift + 0` remain unregistered.

General IME regression checks:

- Normal Jyutping suggestions still rank the same way.
- Reverse lookup trigger keys `r`, `v`, `x`, and `q` still work.
- Candidate paging and numeric candidate selection still use existing input
  counts.
- ABC mode does not eat normal text keys solely because a character variant is
  selected.
- Missing or stale database variant rows leave text unchanged instead of
  failing candidate generation.

## Suggested Implementation Order

1. Extend `Settings.h` and `Settings.cpp` with `CharacterVariant`.
2. Add mapping helpers from `CharacterVariant` to `CharacterStandard`.
3. Add four preserved-key GUIDs and descriptions in `Globals.*`.
4. Register `Control + Shift + 1` through `Control + Shift + 4`.
5. Handle the new preserved-key GUIDs in
   `CCompositionProcessorEngine::OnPreservedKey()`.
6. Add the small `Ime::InputEngine::ConvertText()` wrapper.
7. Convert candidate display text in `AppendInputEngineCandidates()`.
8. Add or defer immediate visible candidate-list refresh after shortcut changes.
9. Build x64 and ARM64 targets.
10. Run the manual verification matrix above.

## Completion Criteria

- The selected character variant is persisted in
  `HKCU\Software\Jyutping\Settings\CharacterVariant`.
- Missing or invalid persisted values fall back to Traditional/Preset.
- `Control + Shift + 1`, `2`, `3`, and `4` select the expected variants from
  the number row.
- Candidate display text is converted according to the selected variant.
- Candidate romanization comments, raw input buffers, and input-count behavior
  are unchanged.
- Selecting a candidate commits the displayed converted text.
- Reverse lookup candidate text follows the selected variant while comments stay
  Cantonese romanization.
- Existing Shift, Shift+Space, and Control+. shortcuts keep working.
- No `TraditionalCharacterStandard` setting is added in this roadmap.
