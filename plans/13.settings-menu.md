Status: Implemented (2026-07-09).


# Settings Menu Roadmap

This plan covers adding a compact settings menu to the existing `Input Mode`
language-bar icon button. The menu should behave similarly to the Rime/Weasel
IME menu: a small native IME menu for frequently changed options, with a
placeholder entry for a future full settings window.

The first menu-backed setting is `CharacterVariant`. The underlying setting,
registry persistence, preserved keys, and candidate conversion behavior already
exist, so this roadmap focuses on exposing the option through the language bar
without changing the input engine model.

## Scope

Add these interactions to the existing `Input Mode` icon button:

- Left-click toggles `InputMethodMode`, preserving the current behavior.
- Right-click opens the IME settings menu.

Add one active settings group:

- `Character Variant`
  - `Traditional`
  - `Hong Kong`
  - `Taiwan`
  - `Simplified`

Add one placeholder command:

- `More Settings...`

`More Settings...` should be visible but disabled or implemented as a no-op in
this roadmap. Opening a dedicated settings page or settings window is explicitly
out of scope.

Out of scope for this roadmap:

- Adding new settings beyond `CharacterVariant`.
- Building a dedicated settings window.
- Changing shortcut behavior for existing preserved keys.
- Changing registry schema or the stored `CharacterVariant` values.
- Changing candidate conversion logic.
- Adding an x86 or ARM64 installer-specific behavior change.

## Current Baseline

Relevant existing Windows files:

- `Jyutping/LanguageBar.*`
  - Owns `CLangBarItemButton`.
  - `OnClick` currently ignores `TfLBIClick` and always toggles the keyboard
    open/close compartment.
  - `InitMenu` and `OnMenuSelect` are present but empty.
- `Jyutping/CompositionProcessorEngine.*`
  - Creates the input-mode language-bar button.
  - Owns `_settings` and `_settingsStore`.
  - Handles preserved-key updates for `CharacterVariant`.
  - Converts candidate display text through `CurrentCharacterStandard()`.
- `Jyutping/Settings.*`
  - Defines `CharacterVariant`.
  - Loads and saves `HKCU\Software\Jyutping\Settings\CharacterVariant`.
- `Jyutping/KeyEventSink.cpp`
  - Refreshes an open candidate list after shortcut-driven
    `CharacterVariant` changes.
- `Jyutping/Resources/*.rc2` and `Jyutping/Resource.h`
  - Own localized user-facing strings.

Existing persisted values must remain unchanged:

```text
HKCU\Software\Jyutping\Settings

CharacterVariant  REG_DWORD  1 = Traditional
                             2 = HongKong
                             3 = Taiwan
                             4 = Simplified
```

## Design Goals

- Keep the language-bar menu native to TSF.
- Keep the generic language-bar button decoupled from IME settings policy.
- Reuse the existing settings model instead of introducing a second state path.
- Show the current `CharacterVariant` with a radio check.
- Apply menu selections immediately.
- Keep the menu structure easy to extend with additional option groups later.
- Preserve existing left-click behavior exactly.
- Avoid implementing a custom popup window for this first menu.

## Proposed Menu Shape

Use a native TSF `ITfMenu` menu populated from
`CLangBarItemButton::InitMenu`.

Suggested structure:

```text
Character Variant >
    Traditional
    Hong Kong
    Taiwan
    Simplified
----------------
More Settings...
```

Menu items under `Character Variant` should use `TF_LBMENUF_RADIOCHECKED` for
the active value. `More Settings...` should use `TF_LBMENUF_GRAYED` until the
dedicated settings UI exists.

## Phase 1: Add Menu-Capable Button Plumbing

1. Extend `CLangBarItemButton` so it can support a menu in addition to the
   current toggle action.
2. Update the language-bar item style to include `TF_LBI_STYLE_BTN_MENU` while
   keeping the existing tray visibility style.
3. Update `CLangBarItemButton::OnClick`:
   - On `TF_LBI_CLK_LEFT`, toggle `GUID_COMPARTMENT_KEYBOARD_OPENCLOSE` as it
     does today.
   - On `TF_LBI_CLK_RIGHT`, return `S_OK` without toggling. TSF should use the
     menu path for the right-click interaction.
4. Keep disabled-button behavior unchanged.

## Phase 2: Add A Narrow Menu Callback Contract

Avoid hard-coding `CharacterVariant` policy directly inside
`CLangBarItemButton`. Add a small callback or owner interface that lets the
button ask the engine:

- what the current `CharacterVariant` is
- to set a new `CharacterVariant`

The implementation can be simple and local, for example:

- a callback pointer plus context pointer, or
- a tiny interface owned by `CCompositionProcessorEngine`.

The callback should not expose candidate-list internals to the button.

## Phase 3: Populate The Settings Menu

Implement `CLangBarItemButton::InitMenu`:

1. Add a `Character Variant` submenu item using `TF_LBMENUF_SUBMENU`.
2. Add stable menu IDs for the four variant commands.
3. Add the four variant entries to the submenu.
4. Mark the active variant with `TF_LBMENUF_RADIOCHECKED`.
5. Add a separator.
6. Add `More Settings...` with `TF_LBMENUF_GRAYED`.

Use localized menu labels, not preserved-key descriptions. For example, the menu
item should be `Traditional`, not `Traditional Characters (Ctrl+Shift+1)`.

## Phase 4: Handle Menu Selection

Implement `CLangBarItemButton::OnMenuSelect`:

1. Map the selected menu ID to a `CharacterVariant`.
2. Call the engine callback to update the setting.
3. Return `S_OK` for known IDs.
4. Return `S_OK` for `More Settings...` while it is disabled/no-op.
5. Ignore unknown IDs safely.

Add engine-side helpers:

```cpp
CharacterVariant CurrentCharacterVariant() const;
void SetCharacterVariant(CharacterVariant variant);
```

`SetCharacterVariant` should:

- update `_settings.characterVariant`
- call `_settingsStore.SaveCharacterVariant(variant)`
- avoid touching unrelated settings

## Phase 5: Refresh Visible Candidates

Menu-driven variant changes should match shortcut-driven behavior as closely as
possible.

Preferred implementation:

1. Extract the candidate-list refresh logic from `CJyutping::OnPreservedKey`
   into a small helper on `CJyutping`.
2. Reuse that helper after menu-driven `CharacterVariant` changes when a
   candidate list is currently visible.

If wiring that refresh from the language-bar menu creates excessive coupling,
the acceptable first version is:

- save and apply the new variant immediately for future candidate rendering
- document that an already-open candidate list refreshes on the next candidate
  update

The preferred version should be used unless it meaningfully complicates the
language-bar ownership model.

## Phase 6: Add Localized Strings

Add resource identifiers and localized strings for:

- `Character Variant`
- `Traditional`
- `Hong Kong`
- `Taiwan`
- `Simplified`
- `More Settings...`

Update:

- `Jyutping/Resource.h`
- `Jyutping/Resources/Strings.en-US.rc2`
- `Jyutping/Resources/Strings.zh-HK.rc2`
- `Jyutping/Resources/Strings.zh-TW.rc2`
- `Jyutping/Resources/Strings.zh-CN.rc2`

Keep shortcut-bearing preserved-key strings unchanged.

## Phase 7: Validation

Build at least:

```powershell
msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=ARM64
msbuild Jyutping.sln /p:Configuration=Debug /p:Platform=x64
```

Manual validation:

1. Left-click the input-mode icon and confirm it still toggles Cantonese/ABC.
2. Confirm the left-click toggle still persists `InputMethodMode`.
3. Right-click the input-mode icon and confirm the settings menu opens.
4. Confirm right-click does not toggle Cantonese/ABC.
5. Confirm the active `CharacterVariant` is radio-checked.
6. Select each variant and confirm the registry value updates.
7. Type candidates and confirm candidate text uses the selected variant.
8. If a candidate list is open, change the variant and confirm candidate text
   refreshes, or document the refresh limitation if the fallback behavior is
   chosen.
9. Confirm `More Settings...` is visible and unavailable/no-op.
10. Confirm localized labels appear under supported UI languages.

## Future Work

Future settings can be added as new groups in the same menu after this roadmap
lands. Good candidates include character width, punctuation form, candidate
preferences, and other frequently changed options.

The dedicated settings page/window should be planned separately. When it exists,
`More Settings...` can be enabled and routed to that UI without changing the
menu-backed `CharacterVariant` behavior.
